At-most-once drops messages on failure. At-least-once reprocesses on failure — fine if your consumers are idempotent. True exactly-once requires idempotent producers, Kafka transactions, and idempotent consumers."}} {{diagram "fund-exactly-once-flow"}} {{think "End-to-end exactly-once needs THREE components working together — most engineers only mention the Kafka side." (whyNot "just idempotent producers alone" "Idempotent producers prevent duplicate writes to Kafka. But if your consumer writes to a DB and crashes after writing but before committing offset, it reprocesses and writes again — duplicate in the DB.") (how "full exactly-once stack" "1) enable.idempotence=true on producer. 2) Use Kafka transactions to atomically write output + commit offset. 3) Consumer writes to DB with upsert on a unique message ID — idempotent consumer.") (whatIf "transaction coordinator broker crashes" "Kafka transactions use a coordinator. If it crashes, incomplete transactions are aborted on recovery. Producers retry with the same producer epoch — Kafka deduplicates by (producer_id, sequence_number).") }} {{code "java" `// Exactly-once producer — Kafka Streams pattern Properties props = new Properties(); props.put("enable.idempotence", "true"); props.put("transactional.id", "my-transactional-producer-1"); props.put("acks", "all"); KafkaProducer producer = new KafkaProducer<>(props); producer.initTransactions(); try { producer.beginTransaction(); producer.send(new ProducerRecord<>("output-topic", key, value)); // Atomically commit: offset + output message producer.sendOffsetsToTransaction(offsets, consumerGroupMetadata); producer.commitTransaction(); } catch (ProducerFencedException e) { // Another instance with same transactional.id took over — abort producer.close(); }`}} {{hint "idempotent consumer pattern" "Store a processed_message_ids table (or use upsert ON CONFLICT DO NOTHING). "How do Kafka Streams achieve exactly-once?" "Kafka Streams sets processing.guarantee=exactly_once_v2 (EOS v2).

{{phase 1 "What Is the Saga Pattern?" "3 min"}} {{say "A Saga is a sequence of local transactions. Each step publishes an event — or sends a command — that triggers the next step. If any step fails, compensating transactions run in reverse to undo prior steps. No global lock, no 2PC coordinator."}} {{diagram "fund-saga-overview"}} {{think "Why not just use a distributed transaction?" (whyNot "2PC (two-phase commit)" "2PC locks rows across services for the duration of the transaction. One slow or crashed service stalls every participant. In microservices across teams, you rarely control all participants.") (how "Saga replaces atomic commit" "Each service commits its own local transaction immediately. Consistency is achieved eventually via compensations, not via a global lock.") }} {{hint "the key insight" "A Saga trades ACID isolation for availability. Two concurrent sagas can see each other's partial state. That's the core tradeoff — you must design your business logic around it."}} {{table "Saga vs Traditional Transaction" (rows (row "Property" "ACID Transaction" "Saga") (row "Scope" "Single DB, global lock" "Multiple services, local commits") (row "Consistency" "Immediate, strong" "Eventually consistent") (row "Failure handling" "Rollback" "Compensating transactions") (row "Scalability" "Blocked by locks" "High — no distributed locks") (row "Complexity" "Simple (DB handles it)" "High — you write compensations") )}} {{triggerQs "Phase 1 Trigger Questions" "When would you choose a Saga over a regular database transaction?" "When the operation spans multiple services that own separate databases. A single-DB monolith should just use a database transaction — Sagas add complexity with no benefit there." "Is a Saga ACID?" "No. Sagas are ACD without isolation. Atomicity is achieved via compensations, consistency is eventual, durability holds per service. The missing I — isolation — is the core gotcha." }} {{phase 2 "Choreography Saga" "5 min"}} {{say "In a choreography Saga, there is no central coordinator. Each service listens to domain events and reacts by performing its local step and emitting the next event. Services are fully decoupled — they don't call each other directly."}} {{diagram "fund-saga-choreography"}} {{think "Choreography is tempting because it's decoupled — but that decoupling creates its own problems." (whyNot "choreography at scale" "With 7 services each reacting to events, tracing a single saga across all of them requires correlating log entries by saga_id across 7 different service logs. There is no single place that shows the current saga state.") (whatIf "an event is published but no consumer is listening" "The saga silently stalls. No orchestrator notices. You need dead-letter queues and saga timeout monitoring to detect this case.") (how "event correlation" "Every event carries a saga_id field. Services must propagate this ID through every step. Observability tooling (Jaeger, Datadog) traces by saga_id.") }} {{code "python" `# Choreography: Order Service publishes event # Payment Service listens and reacts # order_service.py def place_order(order: Order): db.save(order) event_bus.publish("order.created", { "saga_id": order.id, "customer_id": order.customer_id, "amount": order.total, }) # payment_service.py @event_bus.subscribe("order.created") def handle_order_created(event): charge_customer(event["customer_id"], event["amount"]) event_bus.publish("payment.charged", { "saga_id": event["saga_id"], "payment_id": new_payment_id(), }) # restaurant_service.py @event_bus.subscribe("payment.charged") def handle_payment_charged(event): notify_restaurant(event["saga_id"]) event_bus.publish("restaurant.notified", {"saga_id": event["saga_id"]}) `}} {{hint "when to use choreography" "Prefer choreography for simple, short sagas with 2-3 steps. For food delivery (6+ steps with multiple compensations), orchestration wins — you need a single place to see what went wrong."}} {{triggerQs "Choreography Questions" "How do you debug a stalled choreography saga?" "Query the event bus dead-letter queue for the saga_id. Check each service's event log in order. Without a central orchestrator, you're correlating logs — which is why saga_id propagation and structured logging are non-negotiable." "What's the main operational risk of choreography?" "Silent failure. If a service goes down after consuming an event but before publishing the next one, the saga stalls with no alert. You need a saga timeout monitor that queries all in-flight saga_ids older than N minutes." }} {{phase 3 "Orchestration Saga" "5 min"}} {{say "In an orchestration Saga, a central saga orchestrator — a dedicated service — tells each participant what to do via commands. Participants respond with success or failure. The orchestrator tracks the saga state and decides what to do next."}} {{diagram "fund-saga-orchestration"}} {{think "The orchestrator is the source of truth for saga state — that's its strength and its risk." (whyNot "putting orchestration logic in the Order Service" "The Order Service becomes a fat orchestrator responsible for payment, restaurants, and delivery — violating SRP. Saga orchestration should live in a dedicated saga service or be handled by a workflow engine.") (how "orchestrator state machine" "The orchestrator persists a saga_log table with (saga_id, step, status, timestamp). On each response, it transitions the state machine and issues the next command. On crash, it replays from the last committed step.") (whatIf "the orchestrator crashes mid-saga" "On restart, it reads all PENDING sagas from the saga_log and continues from the last successful step. This is why each step must be idempotent — replaying a completed step must be safe.") }} {{compare "Choreography vs Orchestration" (options (best "Orchestration" "Central state tracking, easy debugging, explicit retry logic. "Is the orchestrator a single point of failure?" "Only if stateful and single-instance. A stateless orchestrator with state in Postgres or DynamoDB survives crashes. Run 2+ orchestrator instances with leader election or idempotent workers." }} {{phase 4 "Compensating Transactions" "5 min"}} {{say "Every saga step must have a compensating transaction — the business-level undo. Compensations are not rollbacks: they run after the step committed. They must account for side effects that already happened in the real world."}} {{diagram "fund-saga-compensation"}} {{think "Compensations are harder than they look — they encode business rules, not just DB deletions." (whyNot "DELETE FROM payments WHERE saga_id = X" "A customer's payment may have already settled with Stripe. You can't delete a Stripe charge — you issue a refund. Compensation is a new forward-moving action, not a DB undo.") (whatIf "the compensation itself fails" "Retry the compensation with exponential backoff. If compensation repeatedly fails, page an on-call engineer — this is a stuck saga requiring manual intervention. Log it as a 'compensation failure alert' with full saga context.") (how "compensation ordering" "Compensations run in reverse order of the forward steps. Step 3 compensation runs before step 2 compensation. Each compensation must be idempotent.") }} {{code "python" `# Food delivery saga: 6 forward steps, 6 compensating transactions SAGA_STEPS = [ { "name": "charge_customer", "action": lambda ctx: payment_svc.charge(ctx["customer_id"], ctx["amount"]), "compensate": lambda ctx: payment_svc.refund(ctx["payment_id"]), }, { "name": "notify_restaurant", "action": lambda ctx: restaurant_svc.notify(ctx["order_id"]), "compensate": lambda ctx: restaurant_svc.cancel(ctx["order_id"]), }, { "name": "assign_partner", "action": lambda ctx: delivery_svc.assign(ctx["order_id"]), "compensate": lambda ctx: delivery_svc.unassign(ctx["order_id"]), }, { "name": "pickup_order", "action": lambda ctx: delivery_svc.pickup(ctx["order_id"]), "compensate": lambda ctx: delivery_svc.return_to_restaurant(ctx["order_id"]), }, { "name": "deliver_order", "action": lambda ctx: delivery_svc.deliver(ctx["order_id"]), "compensate": None, # Delivered — no meaningful compensation }, { "name": "pay_partner", "action": lambda ctx: payment_svc.pay_partner(ctx["partner_id"], ctx["fee"]), "compensate": lambda ctx: payment_svc.claw_back(ctx["partner_id"], ctx["fee"]), }, ] `}} {{hint "pivot transactions" "Some steps cannot be compensated after the fact — 'deliver_order' is an example. Don't attempt to 'un-deliver' a physical package."}} {{triggerQs "Compensation Questions" "A customer disputes a charge after the saga completed. Is that a saga concern?" If the step executed but the response was lost, the retry must not double-charge or double-notify. Store (saga_id, step_name) before executing — if it already exists, return the prior result."}} {{code "python" `# Idempotency key pattern for saga steps def charge_customer_idempotent(saga_id: str, customer_id: str, amount: float): idempotency_key = f"{saga_id}:charge_customer" # Check if step already executed existing = db.query( "SELECT result FROM saga_step_log WHERE key = %s", [idempotency_key] ) if existing: return existing["result"] # Return prior result, don't charge again # Execute the actual charge payment_id = stripe.charge(customer_id, amount) # Persist result atomically db.execute( "INSERT INTO saga_step_log (key, result, created_at) VALUES (%s, %s, NOW())", [idempotency_key, {"payment_id": payment_id}] ) return {"payment_id": payment_id} `}} {{think "Idempotency and 'at least once delivery' are inseparable in sagas."

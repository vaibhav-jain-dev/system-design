{{phase 1 "What Is a Circuit Breaker?" "3 min"}} {{say "A circuit breaker wraps calls to an external dependency — a payment service, a database, a third-party API. When the failure rate exceeds a threshold, the circuit opens: all subsequent calls fail immediately without attempting the network call. This protects your service from cascading failures and gives the dependency time to recover."}} {{diagram "fund-cb-overview"}} {{think "The core insight: a slow downstream service is more dangerous than a dead one." (whyNot "just letting failing calls through" "Each call blocks a thread or connection for the full timeout duration (e.g., 30 seconds). 100 concurrent users × 30s timeout = threads exhausted. Your service goes down because a downstream service is slow — not because your code is wrong. This is cascading failure.") (how "circuit breaker prevents cascading failure" "By failing fast (in microseconds instead of 30 seconds), the circuit breaker frees threads immediately. Your service stays responsive for requests that don't need the failing dependency.") (whatIf "the circuit stays open permanently" "The dependency may have recovered but the circuit never closes. Half-open state exists to probe recovery: let one request through, if it succeeds, close the circuit and resume normal traffic.") }} {{hint "the key analogy" "An electrical circuit breaker trips when current exceeds a safe threshold — protecting the wiring from fire. A software circuit breaker trips when failure rate exceeds a threshold — protecting your service from thread exhaustion."}} {{triggerQs "Phase 1 Trigger Questions" "What problem does a circuit breaker solve that a timeout alone doesn't?" "A timeout makes each call fail after N seconds. But if 100 calls/second are hitting a failing service, you still have 100 × N seconds of thread time consumed per second. Circuit breaker stops making the calls entirely — zero thread time consumed during OPEN state." "Is a circuit breaker the same as a retry?" "No. Retry re-attempts a failed call. Circuit breaker stops calling. They're complementary: use retry for transient errors, circuit breaker for sustained failures. Retry without circuit breaker can amplify load on an already-struggling dependency." }} {{phase 2 "State Machine: Closed/Open/Half-Open" "5 min"}} {{say "The circuit breaker is a three-state machine. CLOSED is normal operation. OPEN means the circuit has tripped — all calls fail immediately. HALF-OPEN is the recovery probe state: one request gets through to test if the dependency has recovered."}} {{diagram "fund-cb-state-machine"}} {{table "Circuit Breaker States" (rows (row "State" "Behavior" "Transition Out") (row "CLOSED" "All calls pass through. Count failures." "Failure rate > threshold → OPEN") (row "OPEN" "All calls fail immediately. No network call made." "Wait duration elapsed → HALF_OPEN") (row "HALF_OPEN" "Single test request passes through." "Success → CLOSED; Failure → OPEN") )}} {{think "HALF_OPEN is where most implementation bugs live." (whyNot "letting multiple requests through in HALF_OPEN" "If 10 requests enter simultaneously while half-open, and the dependency is at 50% capacity, you might get 5 successes and 5 failures — ambiguous signal. Let exactly one through. Use an atomic compare-and-swap to ensure only one thread enters HALF_OPEN test mode.") (how "HALF_OPEN transition timing" "After the OPEN state wait duration (e.g., 60 seconds), the circuit transitions to HALF_OPEN automatically. This is timer-driven, not request-driven — the transition happens even if no requests arrive.") (whatIf "the HALF_OPEN test request itself times out" "Treat timeout as failure. Circuit goes back to OPEN for another wait period. A slow response from a recovering service is as dangerous as a fast error — threads are still blocked.") }} {{hint "don't confuse OPEN and CLOSED" "CLOSED = circuit is closed = current flows = normal operation. OPEN = circuit is open = current doesn't flow = all calls blocked. This is electrician terminology — the inverse of what most people expect."}} {{triggerQs "State Machine Questions" "How does the circuit breaker know when to transition from CLOSED to OPEN?" "It maintains a sliding window (count-based or time-based) of recent calls. Count-based: last N calls, X% failed → OPEN. Time-based: in the last N seconds, X% of calls failed → OPEN. Minimum request count prevents tripping on 1 failure out of 2 total calls (50% failure rate but statistically meaningless)." "What happens to requests queued during OPEN state?" "They fail immediately with a circuit-open error — no queue. The caller receives a fast failure and applies its fallback strategy (cached response, default value, error to user). Queuing OPEN-state requests defeats the purpose — you'd still exhaust memory instead of threads." }} {{phase 3 "Configuration Parameters" "4 min"}} {{say "Four parameters control circuit breaker behavior: failure threshold, minimum request count, wait duration in OPEN state, and success threshold in HALF_OPEN. Get these right for each dependency — a payment service and an analytics service have very different sensitivity requirements."}} {{table "Circuit Breaker Parameters" (rows (row "Parameter" "Typical Value" "Effect of Setting Too Low" "Effect of Setting Too High") (row "Failure threshold" "50%" "Trips on transient blips" "Stays CLOSED through real outages") (row "Minimum requests" "20" "Trips after 1 failure (noise)" "Too slow to detect real failures") (row "OPEN wait duration" "60s" "Re-opens immediately, thrashes" "Service recovers but stays OPEN") (row "HALF_OPEN successes" "3" "Closes after one lucky request" "Slow to recover after outage") )}} {{think "There is no universal right configuration — tune per dependency." (how "payment service vs analytics service" "Payment: failure threshold 30%, minimum 10 requests, wait 30s. You need fast detection — money is at stake. Analytics: failure threshold 70%, minimum 50 requests, wait 120s. More tolerance — degraded analytics is acceptable.") (whyNot "a single global circuit breaker config" "A database and a recommendation engine have different latency profiles, failure modes, and business impact. One config will be wrong for at least one of them.") )) {{hint "sliding window type matters" "Count-based windows (last 100 calls) respond faster to sudden spikes. Time-based windows (last 10 seconds) are more representative of current system state. Resilience4j supports both. Choose count-based for high-QPS services, time-based for low-QPS services where count windows may span minutes."}} {{triggerQs "Configuration Questions" "How do you set the minimum request count?" "It should be large enough to be statistically meaningful. With minimum=5, a 60% failure rate triggers on 3 failures — which could be noise. With minimum=20, you need 12+ failures. For a service handling 100 req/s, minimum=20 triggers within 200ms of a real outage. For 1 req/s, use minimum=5 or the circuit never trips." "Should the wait duration in OPEN state be fixed or exponential?" "Start with fixed (60s). Exponential backoff in the OPEN state adds complexity. If the dependency is slow to recover, operators prefer a fixed probe interval so they can predict when traffic resumes. Exponential is better for dependencies that need more time after a restart." }} {{phase 4 "Implementation" "6 min"}} {{say "Here is a Python circuit breaker state machine. In production, use a library — Resilience4j for Java, Polly for .NET, pybreaker for Python — but understanding the implementation tells you exactly what configuration parameters control."}} {{code "python" `import time import threading from enum import Enum from collections import deque class State(Enum): CLOSED = "closed" OPEN = "open" HALF_OPEN = "half_open" class CircuitBreaker: def __init__( self, failure_threshold: float = 0.5, # 50% failure rate min_requests: int = 20, # Minimum calls before tripping wait_duration: int = 60, # Seconds in OPEN state half_open_successes: int = 3, # Successes needed to CLOSE ): self.failure_threshold = failure_threshold self.min_requests = min_requests self.wait_duration = wait_duration self.half_open_successes = half_open_successes self._state = State.CLOSED self._window = deque(maxlen=100) # Sliding window of results (True=success) self._open_since = None self._half_open_successes = 0 self._half_open_in_flight = False self._lock = threading.Lock() def call(self, func, *args, **kwargs): with self._lock: state = self._get_state() if state == State.OPEN: raise CircuitOpenError("Circuit is OPEN — fast failure") if state == State.HALF_OPEN: with self._lock: if self._half_open_in_flight: raise CircuitOpenError("HALF_OPEN test already in flight") self._half_open_in_flight = True try: result = func(*args, **kwargs) self._record_success() return result except Exception as e: self._record_failure() raise def _get_state(self) -> State: if self._state == State.OPEN: if time.time() - self._open_since >= self.wait_duration: self._state = State.HALF_OPEN self._half_open_successes = 0 self._half_open_in_flight = False return self._state def _record_success(self): with self._lock: self._window.append(True) if self._state == State.HALF_OPEN: self._half_open_successes += 1 self._half_open_in_flight = False if self._half_open_successes >= self.half_open_successes: self._state = State.CLOSED self._window.clear() def _record_failure(self): with self._lock: self._window.append(False) if self._state == State.HALF_OPEN: self._state = State.OPEN self._open_since = time.time() self._half_open_in_flight = False return if (len(self._window) >= self.min_requests and self._failure_rate() >= self.failure_threshold): self._state = State.OPEN self._open_since = time.time() def _failure_rate(self) -> float: if not self._window: return 0.0 return 1.0 - (sum(self._window) / len(self._window)) class CircuitOpenError(Exception): pass `}} {{hint "use a library in production" "This implementation omits metrics emission, configurable window types, and thread-safe atomics for HALF_OPEN. Resilience4j (Java), Polly (.NET), and pybreaker (Python) handle all of this. Know the concepts; don't hand-roll it in production."}} {{phase 5 "Fallback Strategies" "4 min"}} {{say "When the circuit is OPEN, you need a fallback — a degraded but functional response.

{{phase 1 "What Is a Circuit Breaker?" "3 min"}} {{say `A circuit breaker wraps calls to an external dependency. When service A calls service B and B is slow:
  • A's threads pile up waiting for B to respond
  • Eventually A runs out of threads and fails too — cascading failure
  • The circuit breaker detects the failure pattern and stops calling B entirely
  • Calls fail fast in microseconds instead of blocking for 30 seconds
`}} {{diagram "fund-cb-overview"}} {{think "The core insight: a slow downstream service is more dangerous than a dead one. A dead service fails fast. A slow service ties up every thread that calls it until the timeout fires." (whyNot "just letting failing calls through" "Each call blocks a thread for the full timeout duration — say 30 seconds. 100 concurrent users times 30s timeout means threads exhausted. Your service goes down because a downstream service is slow — not because your code is wrong. This is cascading failure.") (how "circuit breaker prevents cascading failure" "By failing fast in microseconds instead of blocking, the circuit breaker frees threads immediately. Your service stays responsive for requests that do not need the failing dependency.") (whatIf "the circuit stays open permanently" "The dependency may have recovered but the circuit never closes. Half-open state exists to probe recovery: let one request through, if it succeeds, close the circuit and resume normal traffic.") }} {{hint "the key analogy" "An electrical circuit breaker trips when current exceeds a safe threshold — protecting wiring from fire. A software circuit breaker trips when failure rate exceeds a threshold — protecting your service from thread exhaustion."}} {{triggerQs "Phase 1 Trigger Questions" "What problem does a circuit breaker solve that a timeout alone does not?" "A timeout makes each call fail after N seconds. But if 100 calls/second are hitting a failing service, you still have 100 × N seconds of thread time consumed per second. Circuit breaker stops making the calls entirely — zero thread time consumed during OPEN state." "Is a circuit breaker the same as a retry?" "No. Retry re-attempts a failed call. Circuit breaker stops calling. They are complementary: use retry for transient errors, circuit breaker for sustained failures. Retry without circuit breaker can amplify load on an already-struggling dependency." }} {{phase 2 "State Machine" "5 min"}} {{say `The circuit breaker is a three-state machine:
  • CLOSED — normal operation; calls pass through and failures are counted
  • OPEN — circuit has tripped; all calls fail immediately with zero network attempt
  • HALF_OPEN — recovery probe; exactly one test request gets through
`}} {{diagram "fund-cb-state-machine"}} {{table "Circuit Breaker States" (rows (row "State" "Behavior" "Transition Out") (row "CLOSED" "All calls pass through. Count failures in sliding window." "Failure rate > threshold AND min requests met → OPEN") (row "OPEN" "All calls fail immediately. Zero network calls made." "Wait duration elapsed → HALF_OPEN") (row "HALF_OPEN" "Exactly one test request passes through." "Success → CLOSED; Failure or timeout → OPEN") )}} {{think "HALF_OPEN is where most implementation bugs live." (whyNot "letting multiple requests through in HALF_OPEN" "If 10 requests enter simultaneously while half-open and the dependency is at 50% capacity, you get 5 successes and 5 failures — ambiguous signal. Let exactly one through. Use an atomic compare-and-swap to ensure only one thread enters HALF_OPEN test mode.") (how "HALF_OPEN transition timing works" "After the OPEN state wait duration — say 60 seconds — the circuit transitions to HALF_OPEN automatically. This is timer-driven, not request-driven. The transition happens even if no requests arrive.") (whatIf "the HALF_OPEN test request itself times out" "Treat timeout as failure. Circuit goes back to OPEN for another wait period. A slow response from a recovering service is as dangerous as a fast error — threads are still blocked.") }} {{hint "do not confuse OPEN and CLOSED" "CLOSED means the circuit is closed — current flows — normal operation. OPEN means the circuit is open — current does not flow — all calls blocked. This is electrician terminology, the inverse of what most people expect."}} {{triggerQs "State Machine Questions" "How does the circuit breaker know when to transition from CLOSED to OPEN?" "It maintains a sliding window of recent calls — either count-based (last N calls) or time-based (last N seconds). When failure rate exceeds the threshold AND minimum request count is met, it transitions to OPEN. The minimum request count prevents tripping on 1 failure out of 2 total calls." "What happens to requests queued during OPEN state?" "They fail immediately with a circuit-open error — no queue. The caller receives a fast failure and applies its fallback strategy. Queuing OPEN-state requests defeats the purpose — you would still exhaust memory instead of threads." }} {{phase 3 "Configuration Parameters" "4 min"}} {{say `Four parameters control circuit breaker behavior:
  • failure_threshold — failure rate that trips the circuit
  • min_requests — minimum calls before tripping (prevents noise)
  • wait_duration_in_open — how long to stay OPEN before probing
  • success_threshold_in_half_open — consecutive successes needed to CLOSE
Tune these per dependency — a payment service and an analytics service have very different sensitivity requirements.`}} {{table "Circuit Breaker Parameters" (rows (row "Parameter" "Typical Value" "Too Low Effect" "Too High Effect") (row "failure_threshold" "50%" "Trips on transient blips" "Stays CLOSED through real outages") (row "min_requests" "20" "Trips after 1 failure — noise" "Too slow to detect real failures") (row "wait_duration_in_open" "60s" "Re-opens immediately, thrashes" "Service recovers but stays OPEN too long") (row "success_threshold_in_half_open" "3 consecutive" "Closes after one lucky request" "Slow to recover after outage") )}} {{think "There is no universal right configuration — tune per dependency." (how "payment service vs analytics service parameters" "Payment: failure threshold 30%, minimum 10 requests, wait 30s. Need fast detection — money is at stake. "It should be statistically meaningful. With minimum=5, a 60% failure rate triggers on 3 failures — which could be noise. With minimum=20, you need 12+ failures. "Should the wait duration in OPEN state be fixed or exponential backoff?" "Start with fixed — 60 seconds. Exponential backoff in OPEN state adds complexity. If the dependency is slow to recover, operators prefer a fixed probe interval so they can predict when traffic resumes." }} {{qa "How is a circuit breaker different from a rate limiter?" "A rate limiter caps the number of requests a client can make. A circuit breaker caps the number of requests your service makes to a dependency. Rate limiter protects the server from clients. Circuit breaker protects the client from servers." "Can they work together?" "Yes — rate limit inbound traffic AND circuit-break outbound calls. In a microservice chain A → B → C, A rate-limits its own clients while circuit-breaking its calls to B. B does the same for C."}} {{phase 4 "Implementation" "6 min"}} {{say "Here is a Python circuit breaker state machine. In production, use a library — Resilience4j for Java, Polly for .NET, pybreaker for Python — but understanding the implementation tells you exactly what the configuration parameters control."}} {{code "python" `import time import threading from enum import Enum from collections import deque class State(Enum): CLOSED = 'closed' OPEN = 'open' HALF_OPEN = 'half_open' class CircuitOpenError(Exception): pass class CircuitBreaker: def __init__( self, failure_threshold: float = 0.5, # 50% failure rate trips the circuit min_requests: int = 20, # Minimum calls before tripping wait_duration: int = 60, # Seconds to stay in OPEN state half_open_successes: int = 3, # Consecutive successes needed to CLOSE ): self.failure_threshold = failure_threshold self.min_requests = min_requests self.wait_duration = wait_duration self.half_open_successes = half_open_successes self._state = State.CLOSED self._window = deque(maxlen=100) # Sliding window: True=success, False=failure self._open_since = None self._half_open_success_count = 0 self._half_open_in_flight = False # Atomic gate: only ONE request in HALF_OPEN self._lock = threading.Lock() def call(self, func, *args, **kwargs): with self._lock: state = self._get_state() if state == State.OPEN: raise CircuitOpenError('Circuit is OPEN — fast failure, no network call') if state == State.HALF_OPEN: with self._lock: if self._half_open_in_flight: # Another request is already probing — fail this one fast raise CircuitOpenError('HALF_OPEN probe already in flight') self._half_open_in_flight = True try: result = func(*args, **kwargs) self._record_success() return result except Exception: self._record_failure() raise def _get_state(self) -> State: if self._state == State.OPEN: if time.time() - self._open_since >= self.wait_duration: # Timer expired — probe for recovery self._state = State.HALF_OPEN self._half_open_success_count = 0 self._half_open_in_flight = False return self._state def _record_success(self): with self._lock: self._window.append(True) if self._state == State.HALF_OPEN: self._half_open_success_count += 1 self._half_open_in_flight = False if self._half_open_success_count >= self.half_open_successes: self._state = State.CLOSED self._window.clear() def _record_failure(self): with self._lock: self._window.append(False) if self._state == State.HALF_OPEN: # Failed probe — back to OPEN immediately self._state = State.OPEN self._open_since = time.time() self._half_open_in_flight = False return if (len(self._window) >= self.min_requests and self._failure_rate() >= self.failure_threshold): self._state = State.OPEN self._open_since = time.time() def _failure_rate(self) -> float: if not self._window: return 0.0 return 1.0 - (sum(self._window) / len(self._window)) `}} {{hint "use a library in production" "This implementation omits metrics emission, configurable window types, and slow-call rate tracking.   • Retry handles transient errors — a single packet loss, a brief lock contention
  • Circuit breaker handles sustained failures — a downstream service degraded for 2 minutes
  • Correct layering: retry wraps the individual call; circuit breaker sits at a higher level counting logical request outcomes
`}} {{think "The interaction between retry and circuit breaker is subtle." (how "retry and circuit breaker layering" "Wrap the circuit breaker call with a retry policy.

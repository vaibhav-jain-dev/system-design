{{phase 1 "What Is Consistent Hashing?" "3 min"}} {{say "Consistent hashing solves a specific problem: when you add or remove a server from a cluster, naive modulo hashing remaps almost every key. Consistent hashing remaps only 1/N of the keys — the ones that were owned by the changed node."}} {{diagram "fund-consistent-hashing-overview"}} {{think "Naive modulo hashing: slot = hash(key) % N. Add one server and N changes — every slot recalculates. For 1M keys across 10 nodes, adding an 11th node moves ~909K keys. That's catastrophic." (whyNot "just pre-sharding with a fixed number of slots" "Cassandra does this with 256 vnodes. But you must choose shard count upfront and it still doesn't eliminate rebalancing overhead when nodes change.") (how "consistent hashing reduce moves?" "Place both servers and keys on the same ring (0 to 2³²-1). Each key walks clockwise to the first server token. Add a server: only keys between its predecessor and itself move. Everything else stays.") }} {{hint "hash ring mental model" "Imagine a clock face. Servers sit at 12, 4, and 8 o'clock. A key hashes to 2 o'clock and walks clockwise — it lands on the server at 4 o'clock. Add a server at 3 o'clock: only keys between 12 and 3 o'clock move. That's 1/3 of keys, not all of them."}} {{table "Naive vs Consistent Hashing" (rows (row "Property" "Naive Modulo" "Consistent Hashing") (row "Keys remapped on node add" "~(N-1)/N ≈ 100%" "~1/N") (row "Keys remapped on node remove" "~(N-1)/N ≈ 100%" "~1/N") (row "Load distribution" "Even (if uniform hash)" "Uneven without vnodes") (row "Implementation complexity" "Trivial" "Moderate (sorted ring)") )}} {{triggerQs "Phase 1 Trigger Questions" "Why can't you just pick a large fixed modulo like 1024?" "You can — this is consistent hashing without the ring abstraction. Cassandra does exactly this: 256 virtual partition tokens. But you still have a rebalancing problem when physical nodes join: you must explicitly reassign token ranges. Consistent hashing automates this via ring geometry." "What hash function do you use for the ring?" "MD5 or SHA-1 in legacy systems. MurmurHash3 or xxHash in modern ones — they're faster and have good distribution properties. The key requirement is uniform distribution across the 32-bit or 64-bit key space." }} {{phase 2 "The Ring & Virtual Nodes" "5 min"}} {{say "The ring alone has a load imbalance problem. With three physical nodes, each owns one-third of the ring — but hash boundaries are random, so one node might own 60% and another 10%. Virtual nodes fix this: each physical node is represented by many points on the ring."}} {{diagram "fund-consistent-hashing-vnodes"}} {{think "With V virtual nodes per physical server, the expected variance in load shrinks as V grows. At V=150, empirically the heaviest node carries at most ~10% more than average. At V=10 it can be 50% more." (whyNot "just pick the ring splits manually" "You'd need to rebalance by hand every time a node joins or leaves. Virtual nodes make it automatic — the new node randomly claims V positions and inherits proportional key ownership from every existing node.") (how "virtual nodes work mathematically?" "Physical node P gets V tokens: hash('P#0'), hash('P#1'), ..., hash('P#149'). All V tokens go on the ring as separate ownership points. A key walks clockwise to whichever token it hits first — that token's parent node owns the key.") }} {{hint "V_nodes formula" "Rule of thumb: V = replication_factor × 50. For RF=3, use 150 vnodes per physical node. DynamoDB uses a fixed 64 virtual nodes per physical partition host. "It generates V tokens by hashing its node ID concatenated with an index (node_id + '#' + i). These tokens land at pseudo-random positions on the ring. "No — it minimizes it. You still move 1/N of keys per node change. For 1 billion keys across 10 nodes, adding an 11th still moves 100M keys. That's why DynamoDB pre-shards to many more virtual partitions than physical servers — a physical server owns many virtual partitions, and rebalancing moves whole virtual partitions rather than individual keys." }} {{phase 4 "Implementation" "6 min"}} {{say "A consistent hash ring is a sorted array of (token, node) pairs. To look up a key, hash it and binary search for the first token >= hash(key). If none found, wrap to the smallest token — that's the ring property."}} {{code "python" `import hashlib import bisect from collections import defaultdict class ConsistentHashRing: def __init__(self, virtual_nodes: int = 150): self.virtual_nodes = virtual_nodes self.ring: dict[int, str] = {} # token -> node_id self.sorted_keys: list[int] = [] # sorted token list def _hash(self, key: str) -> int: return int(hashlib.md5(key.encode()).hexdigest(), 16) % (2**32) def add_node(self, node_id: str) -> None: for i in range(self.virtual_nodes): token = self._hash(f"{node_id}#{i}") self.ring[token] = node_id bisect.insort(self.sorted_keys, token) def remove_node(self, node_id: str) -> None: for i in range(self.virtual_nodes): token = self._hash(f"{node_id}#{i}") del self.ring[token] idx = bisect.bisect_left(self.sorted_keys, token) self.sorted_keys.pop(idx) def get_node(self, key: str) -> str | None: if not self.ring: return None token = self._hash(key) idx = bisect.bisect(self.sorted_keys, token) # Wrap around: if past the last token, use the first if idx == len(self.sorted_keys): idx = 0 return self.ring[self.sorted_keys[idx]] def get_nodes(self, key: str, count: int) -> list[str]: """Return `count` distinct physical nodes for replication.""" if not self.ring: return [] token = self._hash(key) idx = bisect.bisect(self.sorted_keys, token) nodes_seen: set[str] = set() result: list[str] = [] for _ in range(len(self.sorted_keys)): node = self.ring[self.sorted_keys[idx % len(self.sorted_keys)]] if node not in nodes_seen: nodes_seen.add(node) result.append(node) if len(result) == count: break idx += 1 return result # Usage ring = ConsistentHashRing(virtual_nodes=150) ring.add_node("node-1") ring.add_node("node-2") ring.add_node("node-3") print(ring.get_node("user:12345")) # e.g. "node-2" print(ring.get_nodes("user:12345", 3)) # ["node-2", "node-1", "node-3"] (replication) `}} {{hint "bisect_left vs bisect" "Use bisect.bisect (= bisect_right) so that if the key hashes exactly to a token, it lands on the NEXT node clockwise. bisect_left would put it on the token itself — either works as long as you're consistent."}} {{think "The ring lookup is O(log N) for binary search on sorted_keys. With 10 nodes and 150 vnodes each, sorted_keys has 1500 entries — bisect finds the slot in ~11 comparisons. Negligible overhead." (how "this scales to millions of keys?" "The ring metadata is tiny: 10 nodes × 150 vnodes = 1500 entries, each an integer and a string. The entire ring fits in a few KB of RAM. It's rebuilt in memory on startup from cluster membership gossip.") }} {{phase 5 "Hotspot Mitigation" "4 min"}} {{say "Consistent hashing distributes keys uniformly — but if one key receives 10x the traffic of all others, virtual nodes don't help. The hot key still maps to one node. The fix is write sharding: split the hot key into N shards and scatter-gather on read."}} {{diagram "fund-consistent-hashing-hotspot"}} {{avoid "Don't say consistent hashing solves hotspot for hot KEYS. It solves hot NODES caused by uneven key distribution across the ring. A single key like 'trending:topic:worldcup' that gets 500K reads/second always maps to one node — consistent hashing doesn't change that."}} {{think "Write sharding for hot keys is a deliberate denormalization. You trade read complexity for write scalability." (how "write sharding works?" "Instead of writing counter to 'trending:worldcup', write to 'trending:worldcup#0' through 'trending:worldcup#N-1' where N=10. Pick shard by random(0,N) or by client_id % N. On read, fetch all 10 shards and sum.") (whatIf "a shard node goes down during a scatter-gather read" "Read from the remaining N-1 shards and extrapolate, or return partial data. Redis Cluster handles this by returning a CLUSTERDOWN error for the failed slot — your client must retry on replica.") }} {{code "python" `# Write sharding for hot keys import random NUM_SHARDS = 10 def increment_counter(redis_client, key: str, amount: int = 1): shard = random.randint(0, NUM_SHARDS - 1) redis_client.incrby(f"{key}#{shard}", amount) def get_counter(redis_client, key: str) -> int: keys = [f"{key}#{i}" for i in range(NUM_SHARDS)] values = redis_client.mget(*keys) return sum(int(v or 0) for v in values) `}} {{triggerQs "Phase 5 Trigger Questions" "How do you detect a hot key before it becomes a problem?"

{{phase 1 "What Is CAP Theorem?" "3 min"}} {{say "CAP theorem states that a distributed system can guarantee at most two of three properties: Consistency (every read returns the most recent write), Availability (every request gets a response), and Partition Tolerance (the system operates despite network splits). In practice, partitions happen — so the real choice is between CP and AP."}} {{diagram "fund-cap-theorem-overview"}} {{think "Brewer's theorem from 2000, proven formally by Gilbert and Lynch in 2002. The insight is that partitions are not optional in real networks — packet loss, switch failures, and datacenter splits happen. You cannot build a distributed system that is immune to partitions." (whyNot "claim your system is CA (consistent + available, no partition tolerance)" "A CA system is a single-node system. The moment you have two nodes that can lose communication, you have a partition. Every distributed database is a P system — the choice is only C vs A during the partition.") (how "a partition manifests in practice?" "Node A and Node B cannot communicate. A write arrives at Node A. Should A accept it? If yes → available but potentially inconsistent (AP). If no → consistent but unavailable (CP). That's the full choice.") }} {{hint "CAP is a trade-off during partitions only" "CAP doesn't say you sacrifice C or A at all times — only during a network partition. When the network is healthy, you can have both. This is why PACELC extends CAP: it covers the latency vs consistency trade-off during normal operation too."}} {{table "CAP Properties Defined" (rows (row "Property" "Formal Definition" "What breaks without it") (row "Consistency (C)" "Every read returns the most recent write or an error" "Stale reads: user sees old data after a successful write") (row "Availability (A)" "Every request receives a (non-error) response" "Requests time out or return errors during partition") (row "Partition Tolerance (P)" "System continues operating despite dropped/delayed messages" "System becomes a single point of failure — any network cut kills it") )}} {{triggerQs "Phase 1 Trigger Questions" "What does 'consistency' in CAP mean? A system can be ACID-consistent but CAP-inconsistent (e.g., read from a replica that hasn't seen the latest write)." "Is CAP theorem still relevant given modern systems?" Both are useful frameworks; start with CAP in interviews, then extend to PACELC if asked." }} {{phase 2 "Consistency vs Availability" "5 min"}} {{say "During a partition, you pick a side. CP systems refuse requests they can't fulfill consistently. AP systems respond with potentially stale data rather than returning an error. The right answer depends on your domain: banking cannot tolerate inconsistency, social feeds can tolerate stale data."}} {{diagram "fund-cap-cp-vs-ap"}} {{think "The business domain dictates the right choice. (whyNot "always choose CP" "Availability is often a hard SLA requirement. A bank's payment processing system that refuses all writes during a 30-second partition has just caused a business incident. "No — proven impossible by the CAP theorem. During a partition, if you require every read to return the latest write (C) and every request to get a response (A), you'd need to synchronize nodes that cannot communicate. Information cannot travel faster than the network. The proof is a contradiction: you'd need to respond with a consistent value without being able to check if another node has a newer value." "What is a 'split-brain' scenario?" "Split-brain occurs in CP systems when both halves of a partition believe they are the active leader and accept writes. This violates consistency. Prevention: quorum — a leader only operates if it can reach a majority. Etcd uses Raft: a candidate must receive votes from (N/2)+1 peers to become leader. A minority partition can never elect a leader, so it halts rather than split-brain." }} {{phase 4 "CP Systems" "4 min"}} {{say "CP systems prioritize consistency: during a partition, they refuse writes they cannot commit to all replicas. ZooKeeper and etcd are the canonical examples — they're designed for low-frequency, high-consistency operations like leader election and distributed configuration."}} {{diagram "fund-cap-cp-systems"}} {{think "ZooKeeper and etcd use consensus algorithms (ZAB and Raft respectively) to guarantee that a write is committed to a majority before acknowledging it. This is strong consistency — but it means write latency is at least one network round-trip to a quorum." (how "Raft achieves CP consistency?" "Every write goes to the leader, which replicates to a quorum of followers before responding. If the leader can't reach a quorum (partition), it steps down. A new leader is elected only if a majority is reachable. Writes halt during leader election (typically 150-300ms in etcd).") (whyNot "use ZooKeeper for high-throughput application data" "ZooKeeper sustains ~10K-100K writes/second. When a transaction commits, Spanner waits for ε before acknowledging, ensuring no future transaction can have an earlier timestamp. This gives external consistency (stronger than linearizability) across global replicas — something no pure software solution achieves." "What happens to a ZooKeeper ensemble during a network partition?" "The minority partition (fewer than (N/2)+1 nodes) stops accepting writes and returns errors. "Eventual consistency means that, given no new writes, all replicas will converge to the same value. It's acceptable when: (1) the data is not financial/inventory, (2) brief staleness doesn't affect user experience, (3) the application can tolerate reading its own write after a short delay. If you want low latency, you acknowledge after writing to the local replica." (how "PACELC classifies systems?" "DynamoDB: PA/EL — partition tolerant + available during partition; low latency (not consistency) during normal operation. "Cassandra's consistency level controls both sides. QUORUM (majority of replicas must ack write AND respond to read) shifts toward PC/EC: writes are slower but reads always reflect recent writes. LOCAL_ONE (fastest, any local replica) is PA/EL. Teams typically use LOCAL_QUORUM — majority within the local datacenter — balancing latency and consistency without cross-region round-trips." "Why does latency matter to consistency at all during normal operation?" "Replication is synchronous if you want strong consistency: the coordinator must wait for all replicas to ack before returning to the client. Each additional replica adds at least one network round-trip. For three replicas in the same datacenter, that's ~1-2ms extra. For three replicas across continents, it's 100-300ms. Async replication (acknowledge before replicas ack) gives low latency but eventual consistency." }} {{phase 7 "Consistency Levels" "5 min"}} {{say "Strong consistency, linearizability, sequential consistency, causal consistency, and eventual consistency form a spectrum.

{{phase 1 "What Is Object Storage?" "3 min"}} {{say `Object storage treats every file as an opaque blob with a key. Unlike a file system — no directories, no in-place updates, no partial writes — just three operations:
  • PUT — write an entire object (atomic, all-or-nothing)
  • GET — read an entire object by key
  • DELETE — remove an object
S3 stores objects in buckets. A bucket is a flat namespace — keys can contain slashes to look like paths, but S3 sees them as opaque strings.`}} {{diagram "fund-s3-overview"}} {{think "Object storage is the right answer whenever you are storing large blobs that are written once and read many times. The 'flat namespace with a key' model is the exact opposite of how databases work — and that's the point: no schema, no index maintenance, no query optimizer needed." (whyNot "storing blobs in Postgres/MySQL" "A 5 MB image stored in Postgres requires: binary escaping, WAL log entry, buffer pool space, replication payload — all for bytes you cannot query. Postgres is optimized for structured rows, not opaque bytes. At 1 million images that is 5 TB of database storage — S3 costs $115/mo, Postgres RDS would cost thousands.") (how "S3 achieves 11 nines of durability" "Every object is automatically replicated across at least 3 Availability Zones within the region. Each AZ is an independent data center with its own power and networking. The probability of losing all three copies simultaneously is astronomically low — 10^-11 per year per object.") }} {{hint "bucket names are globally unique" "S3 bucket names are globally unique across all AWS accounts. my-images is gone once any customer uses it. Use a prefix like acme-prod-media-us-east-1. Include region in bucket names for clarity on cross-region replication."}} {{triggerQs "Phase 1 Trigger Questions" "When would you use S3 vs a database vs a file system?"

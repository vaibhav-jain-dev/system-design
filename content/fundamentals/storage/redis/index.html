{{phase 1 "What Is Redis?" "3 min"}} {{say "Redis is an in-memory key-value store. Sub-millisecond reads. Used as a cache, session store, rate limiter, pub/sub broker, and distributed lock."}} {{qa "Why Redis over Memcached?" "Redis has data structures — sorted sets, lists, hashes, streams. Memcached is just key-value strings. Redis also supports persistence (RDB/AOF), replication, and pub/sub. Memcached is simpler but less capable." "When would you pick Memcached?" "Pure cache, nothing else. Memcached is multi-threaded (Redis is single-threaded), so it can use all CPU cores natively. If you only need GET/SET with massive throughput, Memcached wins on raw performance." }} {{phase 2 "Data Structures" "5 min"}} {{say "Redis isn't just key-value. Its data structures solve specific problems elegantly."}} {{table "Redis Data Structures" (rows (row "Structure" "Operations" "Use Case") (row "String" "GET, SET, INCR, EXPIRE" "Cache, counters, session tokens") (row "Hash" "HGET, HSET, HGETALL" "User profiles, config objects") (row "List" "LPUSH, RPOP, LRANGE" "Message queues, activity feeds") (row "Set" "SADD, SMEMBERS, SINTER" "Tags, unique visitors, social graphs") (row "Sorted Set" "ZADD, ZRANGE, ZRANGEBYSCORE" "Leaderboards, rate limiting (sliding window)") (row "Stream" "XADD, XREAD, XREADGROUP" "Event sourcing, log aggregation") )}} {{thought "Sorted sets are the most underappreciated Redis feature. Rate limiting with sliding window: ZADD timestamp, ZRANGEBYSCORE to count entries in window, ZREMRANGEBYSCORE to clean up. All atomic, all O(log N)."}} {{phase 3 "Caching Strategies" "5 min"}} {{compare "Caching Patterns" (options (best "Cache-Aside (Lazy Load)" "App checks cache → miss → reads DB → writes cache. For URL shortener redirects: 24h TTL is fine. For user profiles: 5-15 min.

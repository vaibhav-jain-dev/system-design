{{phase 1 "What Is Elasticsearch?" "3 min"}} {{say `Elasticsearch is a distributed search and analytics engine built on Apache Lucene. At its core it maintains an inverted index — a mapping from every word to the list of documents containing it:
  • Standard database: "find rows where body contains 'redis'" → O(N) full scan
  • Elasticsearch: look up "redis" in inverted index → O(1) lookup returning document list
The result: full-text search across millions of documents in < 50ms.`}} {{diagram "fund-es-overview"}} {{think "Elasticsearch is optimized for one thing databases are terrible at: searching arbitrary text across large document sets. The inverted index trades write amplification (each word must be indexed) for dramatically faster reads. This is the right trade-off when reads dominate writes." (whyNot "using LIKE '%redis%' in Postgres" "LIKE with a leading wildcard cannot use any index — it forces a full table scan. At 100M rows with 500-byte text each, that's 50 GB scanned per query. Even with a GIN trigram index, Postgres cannot match Elasticsearch for relevance ranking, fuzzy matching, and aggregation performance.") (how "inverted index works" "At index time: tokenize text → normalize (lowercase, remove stopwords, stem) → record {term → [doc_id, position]}. At query time: look up each search term → intersect/union document lists → score by TF-IDF or BM25. The index is stored as a sorted term dictionary + posting lists on disk.") }} {{hint "Elasticsearch vs OpenSearch" "AWS forked Elasticsearch 7.10 in 2021 as OpenSearch after Elastic changed licensing. APIs are near-identical. Use OpenSearch Service on AWS for managed deployment — same mental model as Elasticsearch."}} {{triggerQs "Phase 1 Trigger Questions" "When would you choose Elasticsearch over a relational database for search?" "Choose ES when you need: (1) full-text search with relevance ranking, (2) fuzzy/typo-tolerant matching, (3) aggregations across millions of documents, (4) autocomplete/prefix matching at scale. Keep the relational DB as the source of truth — ES is a derived read index, not a primary store." "What does Elasticsearch NOT do well?" "Transactions — no ACID, no rollback. Strong consistency — indexing is near-real-time (~1s lag). Joins — denormalize data before indexing. Primary storage — data loss possible if index corrupts.   • Fuzzy matching — Levenshtein-based typo tolerance built in; Postgres requires pg_trgm with manual threshold tuning

"BM25 scores each document by: (1) term frequency — how often the search term appears in the document, with diminishing returns for repetition; (2) inverse document frequency — rare terms score higher than common terms; (3) field length normalization — a match in a short title scores higher than the same match in a 10,000-word body. "When business signals matter more than text relevance. Example: product search should boost by sales_rank and recency, not just keyword match. "match tokenizes the query and finds documents containing any of the tokens (OR by default). match_phrase requires all tokens in exact order with no gaps.

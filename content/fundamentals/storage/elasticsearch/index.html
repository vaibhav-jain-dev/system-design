{{phase 1 "What Is Elasticsearch?" "3 min"}} {{say `Elasticsearch is a distributed search and analytics engine built on Apache Lucene. At its core it maintains an inverted index — a mapping from every word to the list of documents containing it:
  • Standard database: "find rows where body contains 'redis'" → O(N) full scan
  • Elasticsearch: look up "redis" in inverted index → O(1) lookup returning document list
The result: full-text search across millions of documents in < 50ms.`}} {{diagram "fund-es-overview"}} {{think "Elasticsearch is optimized for one thing databases are terrible at: searching arbitrary text across large document sets. The inverted index trades write amplification (each word must be indexed) for dramatically faster reads. This is the right trade-off when reads dominate writes." (whyNot "using LIKE '%redis%' in Postgres" "LIKE with a leading wildcard cannot use any index — it forces a full table scan. At 100M rows with 500-byte text each, that's 50 GB scanned per query. Even with a GIN trigram index, Postgres cannot match Elasticsearch for relevance ranking, fuzzy matching, and aggregation performance.") (how "inverted index works" "At index time: tokenize text → normalize (lowercase, remove stopwords, stem) → record {term → [doc_id, position]}. The index is stored as a sorted term dictionary + posting lists on disk.") }} {{hint "Elasticsearch vs OpenSearch" "AWS forked Elasticsearch 7.10 in 2021 as <strong>OpenSearch</strong> after Elastic changed licensing.

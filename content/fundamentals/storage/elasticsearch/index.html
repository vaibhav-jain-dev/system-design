{{phase 1 "What Is Elasticsearch?" "3 min"}} {{say `Elasticsearch is a distributed search and analytics engine built on Apache Lucene. At its core it maintains an inverted index — a mapping from every word to the list of documents containing it:
  • Standard database: "find rows where body contains 'redis'" → O(N) full scan
  • Elasticsearch: look up "redis" in inverted index → O(1) lookup returning document list
The result: full-text search across millions of documents in < 50ms.`}} {{diagram "fund-es-overview"}} {{think "Elasticsearch is optimized for one thing databases are terrible at: searching arbitrary text across large document sets. The inverted index trades write amplification (each word must be indexed) for dramatically faster reads. This is the right trade-off when reads dominate writes." (whyNot "using LIKE '%redis%' in Postgres" "LIKE with a leading wildcard cannot use any index — it forces a full table scan. At 100M rows with 500-byte text each, that's 50 GB scanned per query. Even with a GIN trigram index, Postgres cannot match Elasticsearch for relevance ranking, fuzzy matching, and aggregation performance.") (how "inverted index works" "At index time: tokenize text → normalize (lowercase, remove stopwords, stem) → record {term → [doc_id, position]}. At query time: look up each search term → intersect/union document lists → score by TF-IDF or BM25. The index is stored as a sorted term dictionary + posting lists on disk.") }} {{hint "Elasticsearch vs OpenSearch" "AWS forked Elasticsearch 7.10 in 2021 as OpenSearch after Elastic changed licensing. APIs are near-identical. Use OpenSearch Service on AWS for managed deployment — same mental model as Elasticsearch."}} {{triggerQs "Phase 1 Trigger Questions" "When would you choose Elasticsearch over a relational database for search?" "Choose ES when you need: (1) full-text search with relevance ranking, (2) fuzzy/typo-tolerant matching, (3) aggregations across millions of documents, (4) autocomplete/prefix matching at scale. Keep the relational DB as the source of truth — ES is a derived read index, not a primary store." "What does Elasticsearch NOT do well?" "Transactions — no ACID, no rollback. Strong consistency — indexing is near-real-time (~1s lag). Joins — denormalize data before indexing. Primary storage — data loss possible if index corrupts. Always use ES as a secondary index derived from a primary database." }} {{phase 2 "Core Concepts" "5 min"}} {{say `Four key concepts:
  • Index — like a database table. Contains documents of the same type.
  • Document — a JSON object, the unit of data stored and searched. Like a DB row.
  • Shard — a Lucene index. An index is split into N primary shards for parallelism.
  • Replica — a copy of a shard for fault tolerance and read scaling.
`}} {{diagram "fund-es-sharding"}} {{table "Elasticsearch vs SQL Terminology" (rows (row "Elasticsearch" "SQL Equivalent" "Notes") (row "Index" "Table" "Stores documents of one type") (row "Document" "Row" "JSON object with an _id field") (row "Field" "Column" "Typed: text, keyword, integer, date, geo_point") (row "Mapping" "Schema" "Defines field types; partially dynamic") (row "Shard" "Partition" "Lucene index; fixed at index creation time") (row "Replica" "Read replica" "Copy of primary shard; auto-promoted on failure") (row "Cluster" "Database cluster" "Multiple nodes sharing indices") )}} {{think "Shard count is fixed at index creation time — a critical capacity planning decision." The same condition in a must clause recomputes BM25 scores every time — adding 5-15ms per query."}} {{triggerQs "Relevance & Scoring Questions" "How does BM25 scoring work in Elasticsearch?" "BM25 scores each document by: (1) term frequency — how often the search term appears in the document, with diminishing returns for repetition; (2) inverse document frequency — rare terms score higher than common terms; (3) field length normalization — a match in a short title scores higher than the same match in a 10,000-word body. BM25 replaced TF-IDF as the default in ES 5.0 because it handles long documents better." "When should you use function_score to override default scoring?" "When business signals matter more than text relevance. Example: product search should boost by sales_rank and recency, not just keyword match. Use function_score with field_value_factor on popularity or decay functions on date. Without this, a 5-year-old product with perfect keyword match outranks today's bestseller." "What is the difference between match and match_phrase?" "match tokenizes the query and finds documents containing any of the tokens (OR by default). match_phrase requires all tokens in exact order with no gaps. 'redis cache' as match finds docs with 'redis' OR 'cache'; as match_phrase only docs with the exact phrase 'redis cache' adjacent. Use slop parameter to allow N words between terms." }} {{qa "How does fuzziness work in Elasticsearch?" "Fuzziness uses Levenshtein edit distance — the number of single-character insertions, deletions, or substitutions to transform one string into another. fuzziness: AUTO allows 0 edits for 1-2 char terms, 1 edit for 3-5 chars, and 2 edits for 6+ chars. At query time, ES generates all terms within the edit distance and looks them up in the inverted index — this is why fuzzy queries are 2-5x slower than exact match." "Does fuzziness affect relevance scoring?" "Yes. Fuzzy matches are penalized in the BM25 score. An exact match on 'redis' scores higher than a fuzzy match on 'rediis' → 'redis'. The penalty is proportional to edit distance — 2 edits scores lower than 1 edit."}} {{phase 4 "Indexing Pipeline" "4 min"}} {{say `Elasticsearch is a derived index — never the primary store. The standard pattern:
  • Primary database (Postgres/DynamoDB) → single source of truth
  • Change Data Capture (CDC) stream → Kafka/Kinesis
  • Sync worker → consumes events → indexes into ES
  • Result: ~1 second indexing lag (near-real-time, not real-time)
`}} {{diagram "fund-es-indexing-pipeline"}} {{think "The 1-second refresh interval is the most misunderstood ES behavior." (how "ES refresh works" "ES writes to an in-memory buffer. Every 1 second (default), it flushes to a new Lucene segment and makes it searchable. Documents indexed in the last <1 second are not yet visible to searches. This is 'near-real-time' (NRT) indexing, not real-time.") (whatIf "you need to search a document immediately after indexing" "Call index API with ?refresh=true to force a refresh after the write. Cost: blocks until refresh completes (~50ms). Alternative: ?refresh=wait_for blocks until the next scheduled refresh (~1s). Never set refresh_interval=-1 permanently on a production index.") (whyNot "writing directly to Elasticsearch without a primary DB" "ES does not guarantee durability by default. A node crash can lose up to translog flush_interval (default: 5 seconds) of writes. And ES has no transactions — you cannot roll back a partial batch index. Primary DB + async sync is the correct architecture.") }} {{hint "why CDC over dual writes" "Dual writes (write to DB + write to ES in application code) fail when one write succeeds and the other doesn't — no distributed transaction between Postgres and ES. CDC captures the DB's write-ahead log, so every committed change eventually reaches ES. At 10K writes/sec, even a 0.1% dual-write failure rate means ~860 inconsistencies/day."}} {{hint "refresh_interval tuning for bulk indexing" "During bulk reindex operations, set refresh_interval: -1 to disable automatic refresh. This avoids creating a new Lucene segment every second — reducing I/O by ~40% and speeding up bulk indexing by 2-3x. Re-enable to 1s after the bulk operation completes."}} {{triggerQs "Indexing Pipeline Questions" "How do you handle the lag between primary DB write and ES index update?"

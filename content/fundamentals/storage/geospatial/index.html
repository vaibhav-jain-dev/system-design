{{phase 1 "What Is Geospatial Indexing?" "3 min"}} {{say "A geospatial index lets a database answer proximity queries efficiently — find all restaurants within 2km, or the 10 nearest drivers. Without one, latitude and longitude are just two floating-point columns, and a B-tree index on them cannot answer radius queries without a full table scan."}} {{think "Why can't a standard B-tree answer 'within 5km'?" (whyNot "B-tree on lat/lng" "A B-tree sorts on one dimension. A radius query spans two dimensions simultaneously — you would need to scan every row where lat is in a range AND lng is in a range, then compute actual distance. That is a full table scan on any large dataset.") (how "Spatial index key insight" "A spatial index maps 2D (or 3D) coordinates into a 1D key space where nearby points produce nearby keys. This lets you use a range scan to find candidate rows, then apply an exact distance filter to the small candidate set.") }} {{diagram "fund-geospatial-overview"}} {{table "Geospatial Index Options at a Glance" (rows (row "Index Type" "Structure" "Best For" "Latency") (row "GeoHash" "Base32 string" "Prefix search, cache keys" "< 1ms in Redis") (row "S2 Cells" "64-bit cell ID" "Polygon containment, Google Maps" "< 1ms in memory") (row "H3 Hexagons" "64-bit index" "Aggregation, equal-area zones" "< 1ms in memory") (row "PostGIS GIST" "R-tree variant" "Complex joins, durability" "~10ms at 1M rows") (row "Redis GEO" "Sorted set score" "Real-time nearest-N lookup" "Sub-ms, TTL-aware") )}} {{triggerQs "Opening Questions" "Why can't you just put a regular index on latitude and longitude columns?" "A B-tree index on lat gives you efficient range scans on one dimension, and same for lng, but a proximity query needs both dimensions at once. The database would intersect two index scans and still compute distance on potentially millions of candidates. A spatial index collapses 2D into 1D so a single range scan retrieves a tight candidate set — typically thousands of rows, not millions." "What is the difference between a bounding-box query and a radius query?" "A bounding-box query returns all points within a rectangle (min_lat, max_lat, min_lng, max_lng). It is fast but imprecise — corners of the box are farther than the specified radius. A radius query applies an exact Haversine or ST_DWithin distance filter on top of the bounding-box candidates. Spatial indexes almost always do bounding-box first, then exact filter." }} {{phase 2 "GeoHash" "5 min"}} {{say "GeoHash divides the world into a hierarchical grid. Each cell is encoded as a base32 string. The key insight: two points that share a longer prefix are closer together. This maps geographic proximity to string-prefix proximity — which any key-value store can exploit."}} {{think "GeoHash prefix sharing is powerful but has a critical edge case." (whyNot "pure prefix matching" "Two points separated by only 1 meter can have completely different GeoHash prefixes if they sit on opposite sides of a cell boundary. Always query the 8 neighboring cells in addition to the target cell.") (how "GeoHash precision vs length" "Each additional base32 character roughly halves cell dimensions. A 6-character GeoHash covers ~1.2km × 0.6km. A 9-character GeoHash covers ~4.8m × 4.8m. Choose precision based on acceptable error, not maximum accuracy.") (whatIf "user is exactly on a cell boundary" "Their GeoHash prefix differs from a point 2 meters away across the boundary. The fix: expand your search to all 8 adjacent cells. Redis GEOSEARCH handles this internally — but if you implement GeoHash yourself, always fetch neighbors.") }} {{diagram "fund-geospatial-geohash"}} {{table "GeoHash Precision Reference" (rows (row "Length" "Cell Width" "Cell Height" "Use Case") (row "4" "~39km" "~20km" "City-level bounding") (row "5" "~4.9km" "~4.9km" "Neighborhood search") (row "6" "~1.2km" "~0.6km" "Street-level search") (row "7" "~153m" "~153m" "Building-level search") (row "9" "~4.8m" "~4.8m" "Precise driver location") )}} {{hint "when to use GeoHash as a cache key" "GeoHash strings work directly as Redis or Memcached keys for tiling. "Two points near a cell edge share no common prefix despite being meters apart. Fix: always query the 8 surrounding cells in addition to the target cell. Redis GEOSEARCH handles this automatically. If implementing manually, compute the 8 neighbor hashes using bit manipulation on the GeoHash encoding." "How does GeoHash enable proximity search in a relational database without PostGIS?" "Store the GeoHash string as a VARCHAR column with a B-tree index. For a 5km search at precision-6, compute the target cell and its 8 neighbors, then do a WHERE geohash_6 IN (...9 values...). This is a 9-row index lookup with exact distance post-filter — no full scan." }} {{phase 3 "S2 Cells" "4 min"}} {{say "S2 is Google's geometry library. It projects Earth's sphere onto six cube faces, then recursively subdivides each face into a quadtree. Every cell gets a 64-bit ID. S2's strength is polygon operations — containment, intersection, and covering — which GeoHash handles poorly."}} {{think "S2 is the right tool for polygon containment, not point lookup." (how "S2 hierarchy" "Level 0 is a cube face (~85M km²). Level 30 is ~1cm². For driver matching use level 14 (~0.6km²). For geofencing use level 10 (~100km²). You choose the level based on the smallest feature you need to represent.") (whyNot "GeoHash for geofences" "GeoHash cells are rectangles in lat/lng space — distorted at high latitudes. S2 cells are designed on a sphere and maintain consistent shape globally. Google Maps, Pokémon GO, and Bing Maps use S2 for polygon operations precisely because distortion is minimal.") (whatIf "your polygon spans multiple S2 cells" "S2 computes a cell covering — the minimal set of cells that covers the polygon. Store these cell IDs. To test containment, find the S2 cell for a point and check if any covering cell contains it — O(log N) tree traversal.") }} {{diagram "fund-geospatial-s2"}} {{hint "S2 cell ID is just a 64-bit integer" "Store it in a BIGINT column with a B-tree index. Containment queries become range scans on integer IDs — fast and portable across any database without spatial extensions."}} {{triggerQs "S2 Interview Questions" "Why does Google Maps use S2 instead of GeoHash?" "S2 cells are defined on a sphere with minimal distortion — area and shape remain consistent from equator to poles. GeoHash cells are lat/lng rectangles that become very elongated near the poles. S2 also natively supports polygon covering, which is essential for routing, geofencing, and map tile generation." "How would you implement geofencing for 10,000 delivery zones using S2?" "For each zone polygon, compute an S2 cell covering at level 12 (~300m cells). Store all covering cell IDs in a table: (zone_id, s2_cell_id). For each delivery location, get its S2 cell ID and check parent cells up the hierarchy against the table. One indexed lookup per level — typically 5-10 lookups total to confirm containment." }} {{phase 4 "H3 Hexagonal Grid" "4 min"}} {{say "H3 is Uber's hexagonal hierarchical grid. Every cell at a given resolution has the same area. Hexagons tile perfectly with no gaps and each hexagon has exactly 6 equidistant neighbors — making spatial aggregation uniform. This is why Uber uses H3 for surge pricing zones."}} {{think "Equal-area cells change what's possible for analytics." (how "H3 resolution guide" "Resolution 5 = 252km² per cell (~city district). Resolution 8 is the most common for demand/supply analytics.") (whyNot "rectangles for demand aggregation" "A rectangular grid near the equator covers a different area than the same grid near the poles. Driver supply per cell is not comparable across latitudes. H3 cells at the same resolution always cover the same area, so demand/supply ratios are directly comparable anywhere on Earth.") (whatIf "a surge zone spans multiple H3 cells" "H3 is designed for this — compact and fill operations aggregate child cells into a parent.

How do you prevent double charging a customer?
Idempotency keys stored in Redis with 24h TTL using SET NX. Every write request requires an Idempotency-Key header. On arrival, the API Gateway checks Redis — if the key exists, it returns the cached response immediately without touching the ledger. If the key is new, the payment is processed and the response is stored in Redis before returning to the client.
  • The sentinel value "processing" is stored first (SET NX); replaced with the full response on completion
  • If Redis is down, the PostgreSQL ledger has a UNIQUE(idempotency_key) constraint as a secondary guard
  • 24h TTL covers all mobile client retry windows — no client retries after 24 hours
What if a client accidentally reuses an idempotency key for a different payment amount?
Return 422 Unprocessable Entity with "idempotency_key_reuse". Store a hash of the original request body alongside the response in Redis. On retry, compare hashes — if they differ, reject immediately. Same key must always mean same request.
What is the correct idempotency key format to recommend to clients?
{client_id}:{uuid_v4} where the UUID is freshly generated per payment intent. Never derive the key from the payment amount or timestamp — those can collide. The UUID must come from a cryptographically secure random source to prevent guessing attacks.
Why use the Saga pattern instead of two-phase commit for payment transactions?
Two-phase commit requires all participants to implement the prepare/commit protocol. Bank APIs (Visa, Mastercard) are external black boxes — they expose authorize and void, not prepare and commit. You cannot force Visa to participate in your 2PC protocol.
  • 2PC coordinator is a SPOF: if it crashes after sending prepare but before commit, all participants block indefinitely
  • 2PC is a blocking protocol: participants hold locks until the coordinator decides — catastrophic under contention
  • Saga: each step commits locally and publishes an event; failure triggers compensating transactions in reverse order
What is a compensating transaction and what makes it hard to implement correctly?
A compensating transaction reverses the effect of a prior Saga step — void reverses authorize, release-reserve reverses reserve. The hard part: compensating transactions must be idempotent (the orchestrator retries on failure) and must handle the case where the original step's effect is partially applied.
How does the orchestration-based Saga handle a crash mid-Saga?
The orchestrator persists its state machine in PostgreSQL after each step. On restart, it queries for payments in processing state and resumes from the last confirmed step. Each step is idempotent, so resuming from step N never double-executes step N-1.
How do you ensure the ledger is always balanced?
Double-entry bookkeeping: every payment creates exactly two ledger entries — a debit and a credit of equal magnitude. The invariant SELECT SUM(amount) FROM ledger_entries must always equal exactly 0.
  • PostgreSQL SERIALIZABLE isolation prevents phantom reads — two concurrent payments cannot both read the same balance and both overdraw
  • CHECK (balance >= 0) constraint on the accounts table is a database-level guard against overdrafts
  • Row-level locks acquired in ascending account_id order — deterministic ordering prevents deadlocks
  • A scheduled job runs SELECT SUM(amount) every hour and alerts on any non-zero result
Why SERIALIZABLE isolation instead of READ COMMITTED for the ledger?
READ COMMITTED allows two transactions to read the same balance concurrently, both approve charges, and both commit — leaving the account overdrawn. SERIALIZABLE detects this write-write conflict and aborts one transaction. The application retries the aborted transaction. Serialization failures are rare (<1% under normal load) and retrying is safe because payments are idempotent.
What is the performance cost of SERIALIZABLE isolation at 1000 TPS?
PostgreSQL's SSI (Serializable Snapshot Isolation) implementation tracks read/write conflicts in memory. At 1000 TPS, the overhead is approximately 10-15% higher CPU vs READ COMMITTED. Throughput drops by roughly the same amount. For a payment ledger, this is an acceptable tradeoff — correctness over throughput.
How do you handle a partial payment Saga failure — what if capture succeeds but settlement fails?
Compensating transactions in reverse order, each retried with exponential backoff until successful. If settle fails: retry settle 3× with backoff (1s, 2s, 4s). If all retries fail: move to dead-letter queue, create ops ticket, human resolves within SLA.
  • Capture cannot be compensated with a void after 24h — it must be a refund instead
  • The orchestrator tracks the compensation state separately — if a compensating transaction fails, it needs its own compensation
  • The hardest case: capture succeeded (money moved) but settlement failed (bank not notified). The money is in limbo. A reconciliation job detects this within 24h and re-queues the settlement.
What is the maximum time a payment can be in an unresolved state before it must be escalated?
30 minutes for automated resolution (sweeper retries). After 30 minutes without resolution, an ops alert fires. After 4 hours, a human escalation ticket is created. Card authorizations expire in 7 days — this is the hard deadline for resolving any stuck payment before the auth becomes void.
How do you test Saga failure and compensation in staging?
Chaos injection: a flag in the payment orchestrator can force failure at any step (authorize_fail, reserve_fail, capture_fail, settle_fail). Integration tests cover all 2^N failure combinations for N=4 steps (16 scenarios). Compensation correctness is verified by asserting SUM(ledger_entries) = 0 after each test.
What is tokenization and why is it required for PCI DSS compliance?
Tokenization replaces a real card number (PAN — Primary Account Number) with an opaque, randomly generated token. Merchants store and transmit tokens; the real card number lives only inside a Hardware Security Module (HSM).
  • A breached merchant database contains only tokens — useless to an attacker without the HSM
  • The HSM will process lookup requests but will never export its internal token↔PAN mapping — hardware-enforced
  • PCI DSS requires tokenization or equivalent controls for any system storing card data
  • CVV/CVV2 cannot be stored even encrypted — PCI DSS explicitly prohibits it after authorization
How does the tokenization flow work for a new card entry?
Customer enters card in browser → Stripe.js sends card data directly to Stripe's servers over TLS (never touches merchant's servers) → Stripe HSM stores PAN, returns token → token sent to merchant's backend → merchant stores token for future charges. The merchant's PCI scope is reduced to SAQ A (22 controls) because they never handle raw card data.
What happens to the token if the HSM cluster is permanently lost?
All stored tokens become invalid — merchants would need to recollect card information from all customers. This is a catastrophic data loss scenario. HSMs are operated in geo-redundant active-active clusters with encrypted backups, and have a hardware failure rate of ~0.1% per year. Recovery point objective for HSM data is zero — synchronous replication to the secondary cluster on every write.

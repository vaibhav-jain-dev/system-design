Why choose Postgres over MySQL for the MVP, and at what point should you migrate to DynamoDB?
For an MVP serving <1M users, Postgres wins on developer productivity: it has native JSON support (JSONB), rich indexing (GIN, GiST), and a mature ecosystem. MySQL could work, but Postgres's ACID guarantees with MVCC mean fewer surprises under concurrent writes — critical when you have a single database handling uploads, likes, comments, and feed queries simultaneously. The cost difference at MVP scale is negligible (~$200/mo for an RDS db.r6g.large).

The migration trigger to DynamoDB is not a fixed user count — it's when your write throughput exceeds what a single Postgres instance can handle (~10K-15K writes/sec on a well-tuned instance). For Instagram, this happens around 10-50M users depending on engagement rates. DynamoDB gives you single-digit-ms latency at any scale with on-demand capacity, but you lose joins, transactions across items, and ad-hoc queries. The trade-off is: you pay for eventual consistency (unless you use strong-read, which costs 2x) and must denormalize your data model completely.
What happens if you try to shard Postgres instead of migrating to DynamoDB?
Sharding Postgres is viable but operationally expensive. You need to pick a consistent hashing strategy for shard keys (typically user_id), handle cross-shard queries manually (e.g., a feed query touching 200 users across 16 shards), manage rebalancing when adding shards, and maintain schema migrations across all shards simultaneously. Tools like Citus help, but you're now running a distributed database with Postgres semantics — at that point, DynamoDB's fully managed sharding at $1.25/million writes is often cheaper than the engineering team needed to maintain sharded Postgres.
How does Instagram actually handle this in production?
Instagram famously stayed on sharded Postgres far longer than most companies — they sharded by user_id across thousands of Postgres instances using a custom sharding layer. Their key insight was that most queries are user-scoped (my posts, my feed, my likes), so cross-shard queries are rare. They use logical sharding with a mapping table: a lightweight service maps user_id → shard_id, and each shard is a full Postgres instance. Resharding is done by splitting a shard into two and updating the mapping. This worked because their access patterns are highly localized — but it required a dedicated infrastructure team of 5-10 engineers just for the Postgres layer.
Fan-out-on-write vs fan-out-on-read: what's the basis for the 100K follower threshold?
The threshold comes from a cost-per-post calculation. With fan-out-on-write, each post triggers N Redis ZADD operations (one per follower). At a median post rate of 0.5 posts/day for active users, a user with 100K followers generates 100K × 0.5 = 50K Redis writes/day. A Redis ZADD costs ~0.05ms, so that's 2.5 seconds of Redis time per post — acceptable. But a celebrity with 10M followers would generate 10M writes per post, taking ~500 seconds of Redis time and consuming ~80MB of memory just for that single fan-out. The 100K threshold is where fan-out cost exceeds the cost of a few hundred on-read merges per feed request.

If the threshold is wrong, the impact is asymmetric: too low means more read-time merges (higher P99 feed latency, target <200ms), too high means Redis write amplification and memory pressure. measure and adjust dynamically — some systems use 50K, others 500K, depending on infrastructure costs and read/write ratios.
What about bursty traffic when a celebrity posts — how does that interact with the hybrid model?
Celebrity posts use fan-out-on-read, so the post is written once to the celebrity's post list and merged at read time. But the burst comes from millions of followers opening their feed simultaneously after a push notification. This creates a thundering herd on the celebrity's post list in cache. The mitigation is a token bucket rate limiter on feed requests combined with request coalescing: if 10K requests arrive for the same celebrity's latest posts within 50ms, only one actually hits the cache/database, and the rest wait for that result.
How exactly do you handle the thundering herd on celebrity feed merge without degrading P99 latency?
Three layers: (1) Stale-while-revalidate — serve the previous cached feed while asynchronously merging the new celebrity post, so most users see the post within 2-5 seconds rather than all at once. (2) Probabilistic early expiration — instead of all caches expiring at the same TTL, add jitter (TTL ± 20%) so the thundering herd is spread over ~60 seconds. (3) Single-flight pattern — at the application layer, use a singleflight mutex so only one goroutine/thread fetches the celebrity's post list per cache node, and all other waiters receive the same result. Combined, these keep P99 feed latency under 300ms even during a Beyoncé-posts-at-the-Super-Bowl scenario.
Why use Redis sorted sets for the feed cache instead of Memcached or DynamoDB?
The feed is an ordered, bounded list — you need the latest ~500 post IDs per user, sorted by timestamp/score. Redis sorted sets (ZSET) give you O(log N) insertion, O(log N + M) range queries, and automatic deduplication by member — exactly what a feed needs. Memcached only supports flat key-value blobs, so you'd have to serialize/deserialize the entire feed list on every update (~4KB per user × 500 post IDs), which means read-modify-write race conditions under concurrent fan-out. DynamoDB could work (sort key = timestamp), but at $1.25/million writes and ~500M feed updates/day, you'd pay ~$625/day just for feed writes vs. ~$50/day for Redis cluster memory.

The eviction strategy matters: each user's sorted set is capped at 500 entries using ZREMRANGEBYRANK after each insert, keeping memory bounded. At 500M users with 20% daily active, you need ~100M sorted sets × ~4KB = ~400GB of Redis memory, which fits in a 50-node Redis Cluster (8GB each) with replication.
What happens when a Redis node goes down? How do you avoid cold-start on the feed cache?
Two strategies: (1) Redis Cluster with replicas — each master has 1-2 replicas, so a node failure triggers automatic failover in ~15 seconds with minimal data loss (only the last few seconds of writes if using async replication). (2) Lazy reconstruction — if a user's feed is missing from cache (cold start), the Feed Service fetches the latest 500 posts from the users they follow via a parallel scatter-gather query to the Post Service, merges and sorts them, and populates the cache. This takes ~50-200ms, which is acceptable as a one-time cost. To avoid a mass cold-start after a full cluster failure, we persist Redis snapshots (RDB) to S3 every 5 minutes.
Redis Cluster vs Redis Sentinel — which is the right choice for the feed cache and why?
Redis Cluster is the right choice. Sentinel provides only high availability (automatic failover) for a single master, not horizontal scaling. With 400GB of feed data, you need data sharding across ~50 nodes, which only Redis Cluster provides via consistent hashing over 16,384 hash slots. Cluster also handles rebalancing when adding nodes. Sentinel would require application-level sharding (managing which user maps to which Redis instance), which is the same complexity you're trying to avoid by not sharding Postgres. The trade-off: Cluster has slightly higher operational complexity (slot migrations during scaling) and doesn't support multi-key operations across slots — but feed operations are always single-key (one sorted set per user), so this limitation doesn't apply.
How do you handle like counts at 1M likes/sec on a viral post without melting the database?
Direct COUNT(*) on a likes table would require scanning millions of rows per query — at 1M likes/sec on a viral post, this is a non-starter. Instead, use sharded counters: split the like count for a single post across N counter shards (e.g., 100). Each like increments a random shard: INCR post:{id}:likes:{rand(0,99)}. The display count is the sum of all shards, computed lazily and cached for 5 seconds. This reduces write contention by 100x — instead of 1M writes/sec to one key, you have 10K writes/sec across 100 keys.

For deduplication (preventing double-likes), the source of truth is a DynamoDB table with partition key = post_id and sort key = user_id. The conditional write (PutItem with attribute_not_exists) ensures exactly-once semantics. The sharded counter is eventually consistent — it might briefly show 1,000,003 instead of 1,000,001, but this is acceptable for a social media like count.
Why not just use COUNT(*) with a covering index on the likes table?
Even with a covering index on (post_id, user_id), COUNT(*) on a post with 10M likes requires scanning 10M index entries. On Postgres, this takes ~2-5 seconds because MVCC means the database must check visibility for each row. On DynamoDB, a Query with Select=COUNT still consumes read capacity proportional to item count. The fundamental problem is that COUNT is O(N) in all databases — there's no magic index that maintains a pre-computed count without write amplification. Sharded counters trade perfect accuracy for O(1) reads: reading 100 shards and summing is constant time regardless of total likes.
How do you ensure exactly-once like counting when the user might tap the like button multiple times or retry on network failure?
Three layers of protection: (1) Client-side debounce — the UI disables the button for 500ms after a tap and deduplicates locally. (2) Idempotent write to DynamoDB — the likes table has PK = (post_id, user_id), so duplicate PutItem is a no-op. The conditional expression attribute_not_exists(post_id) returns ConditionalCheckFailedException on duplicates, and we only increment the sharded counter on successful writes. (3) Reconciliation job — a periodic batch job (every 10 minutes) compares the sharded counter sum against the actual COUNT from DynamoDB for hot posts, correcting any drift. This gives us eventually-exact counts: real-time accuracy within ±0.1%, perfect accuracy within 10 minutes. See also how URL shortener handles similar counter challenges at scale.
What's the CAP trade-off for the feed service — do you choose AP or CP, and what are the consequences?
The feed is firmly AP (Available + Partition-tolerant). During a network partition, we continue serving feeds from the local Redis cache even if the Feed Service can't reach the Post Service or Kafka. The worst case is a user sees a stale feed — missing the last 30 seconds of posts — which is far better than showing an error page. Social media feeds have a natural tolerance for staleness: users don't notice if a post appears 5 seconds late, but they absolutely notice a 500 error.

The concrete consistency window is 1-5 seconds under normal operation (Kafka consumer lag + cache propagation) and up to 30-60 seconds during degraded conditions. We chose AP because the business cost of inconsistency (a user sees a post 5s late) is negligible, while the business cost of unavailability (app shows error, user churns) is ~$0.10-$1.00 per incident per user in lifetime value.
What about like counts showing the wrong number — is that also AP, and when does it matter?
Like counts are tunable consistency. For display purposes (showing "1.2M likes"), AP with eventual consistency is fine — the sharded counter is stale by up to 5 seconds. But for the has-the-user-liked-this check (the filled vs. unfilled heart), we need strong consistency because showing the wrong state feels like a bug. This read goes directly to DynamoDB with ConsistentRead=true, bypassing the cache. The cost is 2x read capacity units, but it's a point lookup (single item, <1ms), so the latency and cost impact is minimal. This is a key interview insight: different fields of the same object can have different consistency requirements.
What's the actual consistency window for feed updates, and how can you minimize it without sacrificing availability?
The consistency window breaks down as: Kafka produce latency (~5ms) + consumer poll interval (~100ms) + fan-out write time (~50ms for 500 followers) + client poll/push delay (~1-3s). Total: ~1.5-3.5 seconds end-to-end. To minimize without sacrificing availability: (1) Use WebSocket push instead of polling to cut 1-3s of client delay to ~50ms. (2) Reduce Kafka consumer poll interval to 50ms (trade-off: higher CPU on consumers). (3) For the posting user's own feed, use read-your-own-writes — after uploading, the client optimistically inserts the post into the local feed state before the server confirms fan-out, so the user sees their own post instantly. This brings perceived consistency to <100ms for the poster and <2 seconds for followers, while keeping the system fully AP.

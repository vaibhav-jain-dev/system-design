RRULE Expansion Implementation Deep Dive
{{code "python" `# RRULE examples (RFC 5545) FREQ=DAILY;COUNT=30 # 30 daily occurrences FREQ=WEEKLY;BYDAY=MO,WE,FR # Every Mon/Wed/Fri (infinite) FREQ=MONTHLY;BYDAY=2MO # 2nd Monday of each month FREQ=YEARLY;BYMONTH=1;BYMONTHDAY=1 # New Year's Day # RRULE expansion — on-the-fly within query window def get_events_in_window(calendar_id, window_start, window_end): # 1. Fetch non-recurring events in window directly direct = db.query( "SELECT * FROM events WHERE calendar_id = %s AND start_utc >= %s AND end_utc <= %s AND rrule IS NULL", calendar_id, window_start, window_end ) # 2. Fetch master recurring events that overlap the window masters = db.query( "SELECT * FROM events WHERE calendar_id = %s AND rrule IS NOT NULL AND " "(recur_until IS NULL OR recur_until >= %s) AND start_utc <= %s", calendar_id, window_start, window_end ) expanded = [] for master in masters: rule = rrule.rrulestr(master.rrule, dtstart=master.start_local) duration = master.end_utc - master.start_utc for occurrence in rule.between(window_start, window_end, inc=True): exc = db.get_exception(master.event_id, occurrence.date()) if exc and exc.status == 'CANCELLED': continue # Skip deleted instances if exc and exc.modified_event_id: expanded.append(db.get_event(exc.modified_event_id)) else: expanded.append(materialize(master, occurrence, duration)) return direct + expanded`}} {{think "Pre-expanding vs on-the-fly is the key decision." (whyNot "pre-expand all occurrences into rows" "A weekly event with no end date = infinite rows. Even with COUNT=52, at 2B events/day * 40% recurring * 52 = 41B rows/year. Editing the master requires updating all 52 rows — O(N) write for one user action.") (whatIf "query window is 1 year" "Weekly for 1 year = 52 occurrences at 1μs each = 52μs. Daily for 5 years = 1,825 occurrences, still <2ms.") (how "exception handling" "Two types: EXDATE (cancelled instance) and modified instance. Both in recurring_exceptions table keyed by (master_event_id, occurrence_date). Lookup is O(1).") }}

How do recurring events work without storing N rows for N occurrences?
Store the RRULE string once on the master event row. On each query, expand the RRULE within the time window on-the-fly. Weekly for 1 year = 52 occurrences at ~1μs each — negligible CPU. Exceptions (cancelled/modified instances) live in a recurring_exceptions table keyed by (master_event_id, occurrence_date). During expansion, check exceptions and skip or substitute as needed.
What about editing "this and all following" instances?
Split the master: set recur_until = occurrence_date - 1 day on the original master, then create a new master event starting from the modified occurrence with the updated RRULE. Past occurrences are preserved unchanged. Future occurrences follow the new rule. Both masters remain in the events table with distinct event_ids.
How do you handle infinite recurring events in the free/busy query?
The free/busy query always specifies a bounded time window (e.g., next 7 days). RRULE expansion is bounded by the query window, not the recurrence end. A FREQ=DAILY event with no UNTIL or COUNT expands to exactly 7 occurrences for a 7-day window — O(window_size / frequency), never infinite.
How do you detect scheduling conflicts across all attendees' calendars?
Use the interval overlap predicate: start < requested_end AND end > requested_start. POST /v1/freeBusy takes attendee IDs + time window, queries each attendee's Redis sorted set in parallel via ZRANGEBYSCORE, and merges results. Cache hit >95%. On miss, fall back to PostgreSQL with the covering index.
How do you handle a user with 500 recurring events when checking free/busy?
The Redis free/busy cache stores materialized busy blocks for the next 30 days, pre-expanded from RRULE. Cache is rebuilt asynchronously when a recurring event is created or modified. A user with 500 recurring events has at most 500 * 30 = 15,000 busy blocks for the next 30 days — roughly 1MB per user in Redis. ZRANGEBYSCORE over 15K entries is O(log 15K + M) where M is the result count — under 1ms.
What if an attendee hasn't granted free/busy access to the requester?
Check ACL before querying. If the requester lacks freeBusyReader role on the attendee's calendar, return status="unknown" for that attendee in the freeBusy response. The scheduling UI shows "no information available" for that attendee rather than exposing unauthorized data. This check is against the Redis ACL cache (TTL 5min), not PostgreSQL on every request.
How does multi-device sync stay consistent without a persistent connection?
Delta sync via sync_token (Base64-encoded JSON with calendar_id + last_synced_timestamp). On app open, client sends GET /events?sync_token=... and receives only events changed since then. Tokens expire after 7 days (410 Gone = full sync). Concurrent edits resolved via ETag optimistic locking: If-Match on PUT returns 412 on conflict, forcing re-fetch.
How does the client know deleted events to remove from its local cache?
Soft deletes: events with status=CANCELLED appear in delta sync responses with the cancelled flag set. The client's sync handler checks status and removes CANCELLED events from local storage. Hard deletes would make the event disappear from the delta — the client would never know to remove it. Soft deletes with a 30-day tombstone window guarantee all clients sync the deletion before the row is archived.
Two offline clients both modify the same event — which wins?
Last-write-wins by updated_at timestamp. The second client's PUT with an old ETag gets 412 Precondition Failed. It re-fetches the event, merges (field-level last-write-wins or presents a conflict dialog), and re-submits with the new ETag. For calendar events — which have a single authoritative state — last-write-wins is the correct semantic. Vector clocks would be needed only for CRDTs or collaborative text editing, which calendar fields don't require.
How do you prevent timezone DST bugs in recurring event expansion?
Three rules: (1) store UTC timestamps in the DB, (2) store the IANA timezone name (e.g., "America/New_York"), never the offset, (3) expand RRULE in local timezone, not UTC. Convert master start to local, expand, convert each occurrence back to UTC. A Monday 10am ET event = 14:00 UTC in winter, 13:00 UTC in summer — both correct.
Why is storing the UTC offset wrong even though it looks equivalent?
UTC offsets are point-in-time values. America/New_York is -05:00 in winter and -04:00 in summer. If you store -05:00 and expand a recurring event across the March DST transition, occurrences after the transition are computed 1 hour early in UTC. The IANA timezone database encodes all historical DST transition rules — it knows when each country changes clocks and by how much. UTC offsets encode none of this.
What happens when a country changes its DST rules after events have been created?
The server must use updated tzdata. Events store the IANA timezone name (a stable identifier), not a specific rule version. When tzdata is updated, re-expansion of RRULE automatically uses the new rules. Invalidate the recurring event expansion cache on tzdata version bump. Past events (already-occurred instances) are unaffected — their UTC timestamps are immutable. Only future occurrences are re-expanded with the new rules.
How do you handle sharing permissions at scale without a bottleneck on the ACL table?
Three layers: (1) Redis ACL cache per user (TTL 5 min) handles 95% of auth checks in <1ms. (2) calendar_acl table with index on (user_id) for cache misses. (3) Field-level filtering in the API serializer — DB returns full rows, serializer strips fields based on role.
How do you revoke access immediately (not wait for the 5-minute cache TTL)?
On ACL revocation, invalidate the affected user's Redis ACL cache key immediately: DEL acl:{user_id}. The next request will miss cache and re-read from PostgreSQL, seeing the revocation within milliseconds. For security-sensitive revocations (e.g., employee termination), also issue a token invalidation to force re-authentication — session tokens may still be valid for up to 1 hour otherwise.
How do you handle the case where a shared calendar has 10,000 ACL entries (e.g., a company-wide holiday calendar)?
Don't store 10,000 individual ACL rows. Instead, support group-based ACL: a single calendar_acl row with user_id = group:{domain} or group:{group_id} and role = reader. The authorization check expands group membership from a directory service (LDAP/Google Workspace). Cache group membership per user (TTL 10 min). This reduces a 10K-row ACL to 1 row, making the ACL table O(1) for company-wide calendars regardless of employee count.

How does the trie achieve effectively O(1) lookup at query time?
We pre-compute the top-K suggestions at every prefix node during the hourly rebuild. At query time we simply traverse the trie character by character (O(prefix_length), typically 1-15 chars) and return the cached list at the final node — no subtree traversal needed.
Why not just store a flat hash map of prefix → top-K?
A flat hash map works but uses 2-3x more memory because it stores the full prefix string as each key. The trie shares prefix structure — the path to a node is the key. For 3M prefix nodes, the trie uses ~1.5GB vs ~4GB for an equivalent hash map.
What's the time complexity of building the trie with top-K at each node?
For Q unique queries each of average length L, inserting into the trie is O(Q × L × K log K) — each character position requires updating a bounded top-K heap. At Q=1M, L=8, K=10: ~80M heap operations. In practice, the rebuild completes in 30-60 seconds on 4 vCPU.
How do you handle viral/trending queries that appear mid-cycle before the hourly trie rebuild?
A separate Flink job watches 5-minute tumbling windows on the Kafka query stream. Any query exceeding 10K submissions in a 5-minute window is classified as trending. The job computes a synthetic score and writes it directly to Redis with a 60-minute TTL. Autocomplete service polls Redis for trending overrides and blends them with trie results on every request.
How do you prevent a bot attack from injecting garbage into trending?
Three mitigations: rate limit by IP at CDN (200/min), require valid authenticated uid for query logging (bots without sessions aren't counted), and validate trending candidates with a statistical anomaly check — count must be 3-sigma above the query's own 7-day baseline. A brand-new query spiking from 0 to 10K in 5 minutes fails the baseline check and gets flagged for human review before injection.
What's the lag from a query going viral to appearing in autocomplete?
The Flink 5-minute window closes, the job runs in ~30 seconds, writes to Redis with a 60s TTL refresh cycle. End-to-end: 5 minutes 30 seconds worst case, 5 minutes best case. The CDN TTL for the affected prefix is 60 seconds, so the suggestion propagates globally within 6 minutes of the viral event.
How do you scale to 290K QPS with under 100ms p99?
Three-layer caching reduces origin load by 90%. CDN (CloudFront) caches the top 10K prefixes at the edge — these account for ~88% of traffic. Redis L1 cache handles the next tier. The remaining ~29K QPS hits 50 stateless trie service replicas, each handling 6K QPS with a 1.5GB in-memory trie. Trie lookup is 1-5ms in-process.
What's the memory footprint across the entire fleet?
Each of the 50 trie replicas holds 1.5GB per language trie. For 3 language tries: 4.5GB per replica × 50 replicas = 225GB total fleet memory for trie data. We use r6g.2xlarge instances (64GB RAM) — 3 language tries fit comfortably with room for OS overhead.
How do you autoscale trie services when QPS spikes?
Kubernetes HPA on CPU utilization (target 60%). But trie services take 90 seconds to cold start (download 500MB trie snapshot from S3 + deserialize). We maintain a 30% over-provisioning buffer at all times and pre-scale before known traffic events (e.g., Super Bowl searches). For sudden spikes, the CDN and Redis layers absorb the burst while new replicas warm up.
How does personalization work without violating user privacy?
We never log raw uid+query pairs in the shared query log. The query log contains a daily-salted SHA-256 hash of the uid — this makes cross-day linkage impossible. Personal history lives in a per-uid Redis Hash with 90-day TTL, explicitly scoped to the user's account and deleted on history clear. Personalization blending is ephemeral — the blended results are never stored.
What about GDPR right-to-erasure requests?
A GDPR deletion request triggers: (1) DEL on user:history:{uid} in Redis — immediate, (2) a tombstone record in a deletions table that the Kafka consumer uses to filter out that uid's events from aggregation — effective within the next hourly job, (3) a scan of the S3 query log archives to remove salted-hash occurrences — async, completes within 30 days per GDPR Article 17.
Could you infer a user's uid from their personalized autocomplete results?
No. The personalized suggestions are returned only to the authenticated session that made the request. The response is not cached at the CDN level (we strip uid from the CDN cache key). An attacker observing CDN-cached responses sees only global top-10. The personalized blend is computed at the origin and returned over TLS directly to the authenticated client.
How do you do zero-downtime trie updates — the blue/green swap?
The trie rebuild runs in a background goroutine, building a complete new trie from the aggregated counts into a separate memory region. After sanity checks pass (top queries for 'the', 'google', etc. look correct), we do an atomic pointer swap. In Python, CPython's GIL makes the pointer assignment atomic. In-flight requests using the old trie complete normally — the old trie is reference-counted and freed ~5 seconds after swap.
What if the new trie fails sanity checks?
We abort the swap, keep the current trie serving, and alert on-call. The rebuild job is retried on the next hour. Meanwhile the previous trie snapshot from S3 is available if the current in-memory trie becomes corrupted (e.g., OOM kill forces a cold restart). Two consecutive failed rebuilds trigger a P2 incident.
How do you roll out a new trie to 50 replicas without a thundering herd on S3?
The rebuild job pushes the new trie snapshot to S3, then updates a Redis key trie:latest:{lang} with the new S3 path. Trie replicas poll this key every 5 minutes. When they see a new path, they stagger their S3 download using a random jitter of 0-300 seconds. This spreads 50 downloads over 5 minutes — S3 handles it trivially. No thundering herd.

  • Each node stores its children and the top 10 suggestions already computed for that prefix

A {{algoLink "trie" "Trie"}} is a tree where each node represents one character of a prefix. By pre-storing results at every node, we turn lookup into a simple walk down the tree.

Trie Implementation Deep Dive
{{code "python" `class TrieNode: def __init__(self): self.children: dict[str, TrieNode] = {} self.top_k: list[Suggestion] = [] # pre-computed top-10 at this prefix @dataclass class Suggestion: text: str score: float type: str # "query" | "entity" | "category" class Trie: def __init__(self): self.root = TrieNode() def lookup(self, prefix: str) -> list[Suggestion]: node = self.root for ch in prefix.lower(): if ch not in node.children: return [] # no suggestions for this prefix node = node.children[ch] return node.top_k # O(prefix_length) navigation, O(1) result def build(self, query_counts: dict[str, float]): # Called during trie rebuild job (hourly) for query, score in sorted(query_counts.items(), key=lambda x: -x[1]): self._insert(query, score) def _insert(self, query: str, score: float): node = self.root for i, ch in enumerate(query.lower()): node = node.children.setdefault(ch, TrieNode()) # Maintain top-K at each prefix node self._update_topk(node, Suggestion(query, score, "query")) def _update_topk(self, node: TrieNode, s: Suggestion, k=10): node.top_k.append(s) node.top_k = sorted(node.top_k, key=lambda x: -x.score)[:k]`}}

Storage Estimation Math Deep Dive
  • Average query length: 6 characters, so unique prefix paths go up to depth 6
  • 26^1 + 26^2 + ... + 26^6 = ~3M unique prefix nodes (English, ASCII-only estimate)
  • Each node stores 10 suggestions x 50 bytes = 500 bytes per node
  • 3M nodes x 500 bytes = 1.5 GB for the English trie
  • Non-Latin tries (zh, ja) are smaller — fewer unique prefixes but larger character sets

{{code "python" `# Ranking score formula def compute_score( query: str, base_count: int, # total historical submissions days_since_peak: int, # days since highest traffic day user_affinity: float, # 0.0-1.0 from user history (0 if no history) ) -> float: # Normalize base popularity to 0-1 range using log scale popularity = min(1.0, math.log1p(base_count) / math.log1p(1_000_000)) # Recency boost: queries peak 7 days ago get 50% boost decay recency_decay = 0.5 ** (days_since_peak / 7.0) recency_boost = popularity * recency_decay # Weighted blend score = ( 0.7 * popularity + 0.2 * recency_boost + 0.1 * user_affinity ) return round(score, 4) # Example: # "new york" — 50M historical, peaked 2 days ago, user searched 3x recently # popularity = log(50M)/log(1M) = 1.0 (capped) # recency_boost = 1.0 × 0.5^(2/7) = 0.82 # user_affinity = 0.6 (searched 3 of last 10 queries) # score = 0.7×1.0 + 0.2×0.82 + 0.1×0.6 = 0.764`}}

How does the trie achieve instant lookup?
We pre-compute the top 10 suggestions at every prefix node during the hourly rebuild. At query time, we walk the trie character by character and return the stored list — no subtree search needed.
Why not a flat hash map of prefix to top-K?
A hash map works but uses 2-3x more memory because each key stores the full prefix string. The trie shares prefix structure — the path to a node IS the key. For 3M prefixes: ~1.5GB trie vs ~4GB hash map.
How long does the trie build take?
For 1M queries with average length 8, we do ~80M heap operations to maintain top-K at each node. In practice: 30-60 seconds on 4 vCPU.
How do trending queries appear before the hourly trie rebuild?
A separate Flink job watches 5-minute windows on the Kafka query stream. Queries exceeding 10K hits in 5 minutes are written directly to Redis with a 60-minute TTL. The autocomplete service checks Redis on every request and blends trending results in.
How do you prevent bots from injecting fake trending queries?
Three layers: rate limiting at CDN (200/min per IP), only authenticated sessions count, and a statistical check — the count must be 3x above the query's 7-day baseline. A brand-new query spiking from 0 to 10K gets flagged for human review.
How fast does a viral query appear in autocomplete?
The 5-minute Flink window closes, the job writes to Redis in ~30 seconds, and the CDN TTL is 60 seconds. End-to-end: about 6 minutes from the viral event to global visibility.
How do you scale to 290K QPS under 100ms p99?
Three-layer caching. CDN caches the top 10K prefixes at the edge (~88% of traffic). Redis catches the next tier. The remaining ~29K QPS hits 50 stateless trie servers, each handling 6K QPS with a 1.5GB in-memory trie (1-5ms lookup).
What's the total memory across the fleet?
50 replicas x 4.5GB (3 language tries) = 225GB fleet-wide. We use r6g.2xlarge instances (64GB RAM each), so 3 tries fit easily with room for the OS.
How do you autoscale during a traffic spike?
Kubernetes HPA targets 60% CPU. But cold start takes 90 seconds (S3 download + deserialize), so we keep 30% over-provisioned at all times. For sudden spikes, CDN and Redis absorb the burst while new replicas warm up.
How does personalization work without violating privacy?
We never log raw uid + query pairs. The query log uses a daily-salted SHA-256 hash of the uid, making cross-day linkage impossible. Personal history lives in a per-user Redis Hash with 90-day TTL. The blended results are computed on the fly and never stored.
How do you handle GDPR deletion requests?
Three steps: (1) delete the user's Redis history immediately, (2) add a tombstone so the Kafka consumer filters out their events in the next hourly job, (3) async scan of S3 log archives to remove their hashed entries within 30 days.
Could someone infer a user's identity from their autocomplete results?
No. Personalized results are only returned to the authenticated session over TLS. CDN-cached responses contain only global top-10 (uid is not part of the cache key). An outside observer sees only global results.
How do you do zero-downtime trie updates?
Build the new trie in a background thread. After sanity checks pass (common prefixes return expected results), do an atomic pointer swap. In-flight requests on the old trie finish normally. The old trie is freed ~5 seconds later.
What if the new trie fails sanity checks?
We abort the swap and keep serving the current trie. The rebuild retries next hour. If the server crashes, it downloads the last good snapshot from S3. Two consecutive failures trigger a P2 incident and page on-call.
How do 50 replicas pick up a new trie without overwhelming S3?
The rebuild job writes the snapshot to S3 and updates a Redis key with the new path. Replicas poll every 5 minutes and stagger their S3 downloads with random jitter (0-300 seconds). This spreads 50 downloads over 5 minutes — no thundering herd.

Market Reality: How Indian Startups & Products Solve This
Flipkart
🇮🇳 E-commerce Largest ML recommendation team in India
How they solve it: Two-tower neural network for retrieval (user embedding × item embedding — an embedding is a compact numerical representation of a user or item in a shared vector space, so similar users/items land near each other), gradient boosted trees for ranking. Real-time features via Apache Flink for in-session signals, offline training on Spark for historical patterns.
Stack: Two-tower NN Apache Flink Apache Spark Feast (feature store) Redis (serving)
Key difference: Model trains on purchase intent signals (add to cart, search query, time on page), not just purchase history — capturing mid-funnel intent that purchase-only models miss. This improves CTR (Click-Through Rate — the percentage of users who click on a recommendation out of those who saw it) by 15% because a user who spends 3 minutes on a product but doesn't buy is a stronger signal than a one-click impulse purchase.
{{caveat "Two-tower models require separate training of user and item towers — if you couple them, you cannot precompute item embeddings offline, which destroys serving latency. Interviewers expect you to explain why the towers must be independent."}}
Meesho
🇮🇳 Social Commerce Social-proof-boosted recommendations
How they solve it: Apache Spark for offline collaborative filtering (CF — a recommendation technique that finds patterns across users' behaviors: "users who bought X also bought Y"), Feast feature store for real-time feature serving, Redis for real-time social graph lookups. Recommendations include "X of your WhatsApp contacts bought this" — combining collaborative filtering output with social graph data.
Stack: Apache Spark Feast Redis (social graph) Collaborative filtering
Key difference: Social trust layer on top of collaborative filtering — requires a Redis social graph lookup at serving time, adding a "friends who bought this" signal unique to social commerce. In markets where WhatsApp is the primary shopping discovery channel, this social signal outperforms personalization signals from browsing history alone.
Nykaa
🇮🇳 Beauty E-commerce Visual + collaborative filtering hybrid
How they solve it: CNN-based visual similarity for same product family across brands, user-based collaborative filtering for personalization, and a knowledge graph for product chemistry relationships (e.g., Vitamin C serum → sunscreen regimen pairing).
Stack: CNN (visual similarity) Collaborative filtering Neo4j (knowledge graph) Spark
Key difference: Product chemistry relationships in a Neo4j knowledge graph capture "regimen" patterns (buy Vitamin C serum → recommend sunscreen) that pure co-purchase signals miss. Co-purchase is noisy — users buy many unrelated items in one basket. A domain knowledge graph encodes causality that purchase data alone cannot surface.
{{goodToKnow "Knowledge graphs for recommendations are most effective in domains with strong item-to-item causal relationships (beauty regimens, recipe ingredients, medication interactions). For general e-commerce without domain-specific item relationships, collaborative filtering alone outperforms knowledge-graph-augmented approaches."}}
Netflix
🌐 Global Reference Industry-defining two-stage pipeline
How they solve it: Two-stage pipeline (candidate generation → ranking), A/B testing framework at every layer, contextual bandits for homepage layout optimization. Personalizes not just content but the thumbnail image shown — the same movie has different cover images for different users based on viewing history.
Stack: Two-stage pipeline Contextual bandits Metaflow (ML workflows) Feast (feature store)
Key difference: Recommendation system on top of a recommendation system — personalizing the thumbnail image is a second-order personalization layer that increases click-through without changing the underlying content ranking. Indian streaming platforms (Hotstar, JioCinema) copy this pattern exactly for cricket match thumbnail selection.
{{caveat "Cold start is the single most common interview follow-up for recommendation systems. New users have no interaction history, new items have no engagement data. You must have a concrete answer: content-based features for new items, popularity-based fallback for new users, and contextual bandits (an explore/exploit algorithm — 'exploit' shows items known to work well, 'explore' tries new items to gather signal; the algorithm balances both automatically) to bootstrap signal."}}
OSS & Managed Alternatives
Surprise — Python collaborative filtering library (SVD, KNN, matrix factorization)
LightFM — hybrid matrix factorization (content + collaborative signals)
Feast — open source feature store, widely used in Indian ML stacks
TensorFlow Recommenders — Google's two-tower and retrieval framework
Metaflow — Netflix's ML workflow orchestration, open source
Apache Mahout — distributed collaborative filtering (legacy, Spark-based)
AWS Personalize — managed recommendation API, handles cold-start automatically
Google Vertex AI Recommendations — managed, retail-optimized recommendation engine
Azure Personalizer — managed contextual bandit service
{{goodToKnow "Feast (feature store) appears in nearly every production recommendation stack because it solves the training-serving skew problem — features computed offline for training must be served identically at inference time. Without a feature store, teams often compute features differently in batch (Python/Spark) vs online (Java/Go), causing silent accuracy degradation."}} {{caveat "A/B testing recommendation models is not a simple 50/50 traffic split. Recommendation quality has network effects — showing worse recommendations to the control group can permanently change their behavior (they stop clicking), making the experiment non-reversible. Interleaving (mixing results from both models in one list) is the industry-standard solution."}} {{mustKnow "Two-stage pipeline (candidate generation → ranking) is the universal architecture — candidate generation reduces millions of items to ~1000 candidates in O(1) via ANN (Approximate Nearest Neighbor — fast vector similarity search that finds close matches without scanning every item), ranking scores those 1000 with an expensive model" "Feature stores (Feast) solve training-serving skew by ensuring the same feature computation logic is used in both batch training and real-time serving" "Collaborative filtering (CF) finds similar users/items from interaction patterns; content-based filtering uses item metadata — hybrid systems combine both via two-tower models or LightFM" "Cold start requires three separate strategies: popularity fallback for new users, content-based features for new items, and contextual bandits (explore/exploit) to bootstrap signal"}}
Market verdict: Indian e-commerce companies (Flipkart, Myntra, Nykaa) invest heavily in custom recommendation systems because product discovery drives 30–40% of revenue. The tech stack converges on Spark (offline) + Flink (real-time) + Feast (features) + Redis (serving) — the same stack Netflix and Airbnb use. The India-specific insight: social commerce (Meesho) adds a social trust layer on top of standard collaborative filtering, which is unique to markets where WhatsApp is the primary shopping discovery channel.

How do you absolutely guarantee no double-booking at 100K concurrent users?
Redis SET NX is atomic at the key level — guaranteed by Redis's single-threaded command execution. Two concurrent SETNX on the same key: one returns 1 (acquired), one returns 0 (rejected). There is no race condition possible within a single Redis instance. With Redlock across 3 nodes, the lock is safe against single-node failures. Final safety net: Postgres SERIALIZABLE transaction on confirm prevents any write conflict that somehow slipped past Redis.
What if the Redis lock says "available" but Postgres says "booked"?
This can happen if the lock key expired (10min TTL) but the Postgres booking record was committed. This means the seat is booked in Postgres — Postgres is the source of truth. The confirm handler always re-reads Postgres: if a confirmed booking for those seat_ids + show_id already exists, it rejects the new confirm. The Redis stale state is irrelevant for post-lock decisions.
Why not use Postgres SERIALIZABLE transactions alone — do you really need Redis?
SERIALIZABLE transactions serialize conflict detection at commit time. During a rush, 10K concurrent transactions all trying to lock the same seat would queue on the same row — Postgres serializes them, first wins, rest abort. The retry storm under high contention causes 100K-seat-request load on Postgres. Redis SETNX handles 10K concurrent attempts in O(1) per attempt, O(log N) throughput — no retry storm, no DB connection exhaustion. Redis is the fast pre-filter; Postgres is the durable ledger.
Walk me through exactly what happens when 100K users hit "Book Now" for a show with 500 seats.
Step 1: All 100K requests hit CloudFront (browsing state). Step 2: ~100K simultaneous POST /bookings/lock requests arrive at ALB — routed to Booking Service instances (auto-scaled). Step 3: Each request tries SET NX on its seat keys in Redis. ~500 succeed (one per seat). ~99,500 get 0 back and return 409. Step 4: 500 successful locks enter the payment flow. Step 5: 15% (~75) of those abandon — keys expire at 10 minutes, seats re-enter pool. Net result: 425 confirmed bookings, 75 seats recycled to waitlist.
How does the system handle 100K simultaneous lock requests without crashing?
Booking Service is stateless — horizontally scaled to 50+ ECS tasks at peak. Each task processes requests independently, hitting Redis directly. Redis handles 200K+ SET NX commands/second on a single instance. With a 6-node cluster, theoretical throughput is 1.2M ops/sec. 100K concurrent requests arriving over 1-2 seconds = ~50K–100K ops/sec peak. Well within capacity.
How do you prevent the 99,500 rejected users from retrying in a loop and amplifying load?
On 409 response, the client shows "Seats Unavailable — try other dates" and does NOT auto-retry. Exponential backoff with jitter applies only if the user manually selects different seats. Rate limiting at the API gateway: 10 lock attempts per user per show per 5 minutes. This converts the retry storm into at most 10x the initial load — manageable vs unbounded retries.
What happens if the payment webhook arrives after the seat lock expires?
Three sub-cases. (1) Lock expired and seat is still unbooked by anyone — proceed to confirm, write to Postgres, lock was just a reservation mechanism. (2) Lock expired and seat was re-locked by User B but not yet confirmed — decline User A, refund User A, don't disturb User B's pending booking. (3) Lock expired and seat is fully confirmed for User B — decline User A, refund User A immediately via gateway API. The payment webhook handler re-checks all three states atomically before writing to Postgres.
How do you prevent double-refunds if the webhook fires twice (network retry)?
Idempotency on the payment record: payments table has a UNIQUE constraint on gateway_txn_id. The INSERT on payment confirmation will fail with a duplicate key error on the second webhook. Wrap in a Postgres upsert or check-before-insert. Return 200 to the gateway on duplicate (don't return 409 or the gateway will keep retrying).
If a user's payment succeeds but our confirm webhook handler crashes mid-execution, what state are we in?
Booking is still in 'locked' state in Postgres. Payment is 'success' in the gateway. Redis lock has TTL. Recovery: a reconciliation job runs every 10 minutes scanning bookings in 'locked' state where payment is 'success'. It re-runs the confirmation flow. This is idempotent — the second run sees payment already succeeded, writes confirmed status. User gets their ticket, slightly delayed.
How does real-time seat availability work without overwhelming the database?
Read path layered in three tiers: (1) WebSocket push via Redis Pub/Sub — every lock and release publishes to seat:show:{id} channel. All seat map subscribers receive updates in < 10ms. No DB reads. (2) Redis cache — GET /shows/{id}/seats reads from Redis with 5s TTL. Cache is invalidated on lock/release events. 90% cache hit rate. (3) Postgres read replica — fallback if Redis cache miss. Runs the seat availability query with the index on bookings(show_id, status). ~10ms query time on a warm index.
What if two users see the same seat as "available" simultaneously and both try to lock it?
This is the normal case — the seat map is eventually consistent (5s cache TTL). Two users seeing the same seat as available is expected and correct. The lock is the serialization point, not the seat map view. First SETNX wins. Second gets 0 (unavailable). The seat map updates in < 5 seconds via Pub/Sub. This is not a bug — it's the intended design.
Why is eventual consistency acceptable for the seat map view but not for the lock?
The seat map is a hint to help users choose — showing a seat as briefly available when it's actually locked is harmless; the worst case is the user picks a seat and gets a 409 on lock. The lock must be strongly consistent because it's the gate to actual payment processing. Two different consistency requirements get two different storage systems: Redis (strong, atomic) for locks, Redis cache (eventual, cached) for display state.
How would you extend this system to handle 50M-seat events (e.g., an IPO ticket lottery)?
At 50M concurrent applicants for 10K seats, the Redis lock model breaks — 50M SETNX ops simultaneously would saturate even a large Redis cluster. Switch to a lottery model: collect all applications in a 30-minute window (write to Kafka, no locks). Run a batch lottery draw after window closes — randomly select 10K winners from the applicant pool. Notify winners with a 4-hour payment window. This converts a real-time contention problem into a batch selection problem, scalable to any number of applicants.
Doesn't the lottery model remove the "first-come first-served" fairness perception?
Yes, intentionally. First-come-first-served at internet scale is actually "whoever has the best internet connection and fastest device" — not truly fair. A 30-minute application window levels the playing field. This is how IPO applications, concert lotteries (Wimbledon, Super Bowl), and immigration lottery systems work at national scale.
How do you implement the lottery draw fairly and verifiably?
Use a cryptographically secure random seed (e.g., SHA-256 of the previous Bitcoin block hash + application window close timestamp). Apply Fisher-Yates shuffle to the applicant list using this seed. Select first N as winners. Publish the seed publicly so any applicant can verify the draw was not manipulated. This is the same technique used by most national lotteries for verifiable fairness.

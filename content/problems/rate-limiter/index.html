Why is Sliding Window Counter preferred over Token Bucket for most production rate limiters?
Sliding Window Counter provides the best balance of accuracy and simplicity. It uses only O(1) memory per key (two counters: current window + previous window), eliminates the boundary burst problem of Fixed Window, and requires no complex state management like Token Bucket's floating-point token tracking. Cloudflare and Stripe both use variants of this algorithm. The weighted average calculation prev_count × weight + curr_count achieves within high accuracy of a true sliding window log while using constant memory.
When would you choose Token Bucket over Sliding Window Counter?
Choose Token Bucket when your API explicitly wants to allow controlled bursts. For example, AWS API Gateway uses Token Bucket because API consumers often need to send a batch of requests quickly (e.g., uploading 10 images at once), then slow down. The bucket capacity controls burst size while refill rate controls sustained throughput. This is a deliberate product decision, not a technical one.
How do you prevent Token Bucket from being exploited by saving up tokens?
Set capacity = refill_rate × small_window (e.g., 2× the per-second rate). This means a user can burst at most 2 seconds worth of requests. Also consider dual-rate limiting: Token Bucket for burst control + a Sliding Window for sustained rate, and the stricter limit wins. See: Distributed Rate Limiting for multi-layer strategies.
What is the exact formula for Sliding Window Counter accuracy?
The error bound is |estimated - actual| ≤ (prev_window_count × elapsed_fraction / window_size). In practice, the maximum error occurs when traffic is perfectly periodic. For random traffic patterns (which is the real world), the error approaches zero. Cloudflare published benchmarks showing error well under 1% at 10K RPS.
How do you handle rate limiting in a multi-region deployment without adding latency?
The three approaches are: per-region limits (simplest, user gets N req/region), async gossip sync (regions sync counters every 1-5 seconds, used by Cloudflare), and global Redis (single source of truth, +50-100ms latency). The recommended approach is async gossip sync: each region maintains a local Redis with accurate local counts, and a background process propagates deltas to other regions. The trade-off is a 1-5 second window where a user could exceed the global limit by up to N × regions.
Why not use Redis Global Datastore for cross-region consistency?
Redis Global Datastore provides async replication (not sync), so you still get eventual consistency. It also adds complexity: you need to handle conflict resolution for INCR operations (last-writer-wins can lose counts). Custom gossip sync is simpler because you control the merge strategy: SUM deltas rather than last-writer-wins.
How does Cloudflare's actual multi-region rate limiter work?
Cloudflare uses a hierarchical approach: each PoP (200+ locations) has local counting, PoPs in the same region aggregate to a regional counter every ~1s, and regional counters sync globally every ~5s. They accept that a determined attacker can get ~3× the limit for a few seconds, but their DDoS mitigation layer catches volumetric attacks before the rate limiter is even involved. See: Edge Cases for DDoS handling.
How do you test multi-region rate limiting accuracy?
Use chaos engineering: inject artificial latency between regions (tc netem), simulate region failures, and measure over-limit percentage. Key metric: "over-limit ratio" = requests allowed above the configured limit / total requests. Target: <5% over-limit during normal operation, <20% during region failover.
Why use Redis Lua scripts instead of distributed locks for atomic rate limiting?
Lua scripts execute atomically on the Redis server — no other commands run during execution. This eliminates race conditions without locks. A distributed lock (e.g., Redlock) would add 2-3 extra round trips (acquire + release + fencing), increasing latency from <1ms to 5-10ms. Locks also introduce deadlock risk, require timeouts, and reduce throughput due to contention. Lua scripts are the correct primitive for rate limiting because the operation is simple and short-lived.
What happens if a Lua script takes too long in Redis?
Redis has a lua-time-limit config (default 5s, we set 100ms for rate limiting). If exceeded, Redis starts accepting read-only commands but the script continues until completion. It does NOT kill the script — it only unblocks other reads. For rate limiting Lua scripts, execution should be <0.1ms because we're doing 2-3 simple operations. If your Lua script approaches 100ms, you have a design problem.
Can EVALSHA improve performance over EVAL?
Yes. EVALSHA sends only the SHA1 hash of the script (40 bytes) instead of the full script text (~500 bytes). On first call, use SCRIPT LOAD to cache the script. Subsequent calls use EVALSHA with the hash. This reduces network bandwidth by ~92% per rate limit check. At 500K checks/sec, that's saving ~220 MB/sec of network traffic. All production rate limiters should use EVALSHA.
How do you handle Redis cluster mode with Lua scripts?
Redis Cluster requires all keys in a Lua script to hash to the same slot. Use hash tags: rl:{user:123}:endpoint — the {user:123} part determines the slot. This ensures all rate limit operations for a user go to the same shard. Caveat: if one user generates disproportionate traffic, their shard becomes a hot key problem.
How should the rate limiter behave when Redis is completely down?
Implement a circuit breaker pattern. After N consecutive Redis failures (e.g., 3), trip the circuit to "open" state: bypass Redis entirely and either fail-open (allow all traffic) or fail-closed (reject all) based on the endpoint's configured policy. After a cooldown (e.g., 30s), enter "half-open" state: send one probe request to Redis. If it succeeds, close the circuit and resume normal operation.
Can you do rate limiting without Redis at all, as a fallback?
Yes, use local in-memory counters as a degraded mode. Each app server maintains its own token bucket with limit / num_servers capacity. This is less accurate (each server allows its fraction independently) but provides basic protection. Combine with consistent hashing at the load balancer to route the same user to the same server, improving accuracy of local counters.
What about using DynamoDB atomic counters as a Redis alternative?
DynamoDB UpdateItem with ADD is atomic and durable. Latency is 5-10ms (vs Redis <1ms), which is too high for hot-path rate limiting. But it works well for low-rate limits like login attempts (5/15min) where 5ms extra is negligible. DynamoDB also gives you built-in TTL for automatic key expiration. Cost: $1.25 per million writes vs Redis flat rate — cheaper at low scale, expensive at high scale. See: Cost Estimates for comparison.
How do you monitor rate limiter health?
Key metrics: rate_limit_decisions_total (counter, labels: allowed/rejected), rate_limit_latency_ms (histogram, p50/p99), redis_connection_errors (counter), circuit_breaker_state (gauge: closed/open/half-open). Alert on: rejection rate spike (>10% of traffic), Redis latency >5ms p99, circuit breaker tripping. Dashboard should show per-endpoint and per-tier breakdowns.
How does rate limiting differ for WebSocket connections vs HTTP APIs?
WebSockets have two rate limiting dimensions: connection rate (new connections/min) and message rate (messages/sec per connection). For connection rate, use standard Sliding Window Counter at the HTTP upgrade endpoint. For message rate, use Token Bucket per connection because WebSocket messages are inherently bursty (e.g., typing indicators, real-time updates). The token bucket allows short bursts while capping sustained throughput.
Where do you enforce WebSocket message rate limits — server-side or client-side?
Both. Client-side: implement backpressure with a message queue and send rate. Server-side: Token Bucket in the WebSocket handler. If the server rejects a message, send a rate_limit frame back to the client with retry-after info. Unlike HTTP 429, there's no standard WebSocket rate limit frame, so define it in your protocol (e.g., JSON {"type":"rate_limited","retry_ms":500}).
How do gaming companies handle rate limiting for real-time multiplayer?
Gaming uses leaky bucket per player per action type. Movement updates: 60/sec (tick rate). Chat: 1/sec. Ability use: per-cooldown. The rate limiter is embedded in the game server (not external Redis) for <0.01ms overhead. Cheating detection layers above the rate limiter look for impossible action sequences. This intersects with distributed counting in the multiplayer server mesh.
How do you rate limit GraphQL APIs where one request can be cheap or expensive?
GraphQL needs query complexity scoring instead of simple request counting. Assign a cost to each field/resolver (e.g., user=1, user.posts=10, user.posts.comments=50). Sum the query cost and deduct from a Token Bucket. GitHub's GraphQL API uses this: 5000 points/hour, each query costs variable points. This prevents abuse via deeply nested queries while allowing many cheap queries.

Returns (op1', op2') where: - op1' can be applied after op2 - op2' can be applied after op1 Both paths reach the same final document state. """ op1_prime, op2_prime = [], [] i1, i2 = 0, 0 while i1 < len(op1) or i2 < len(op2): op_a = op1[i1] if i1 < len(op1) else None op_b = op2[i2] if i2 < len(op2) else None if op_a and op_a.get('insert'): # Insert in op1 → shift op2's position by len(insert) op1_prime.append(op_a) op2_prime.append({'retain': len(op_a['insert'])}) i1 += 1 elif op_b and op_b.get('insert'): # Insert in op2 → shift op1's position by len(insert) op2_prime.append(op_b) op1_prime.append({'retain': len(op_b['insert'])}) i2 += 1 elif op_a and op_a.get('retain') and op_b and op_b.get('retain'): # Both retain — take the minimum min_len = min(op_a['retain'], op_b['retain']) op1_prime.append({'retain': min_len}) op2_prime.append({'retain': min_len}) # Adjust remaining lengths op1[i1] = {'retain': op_a['retain'] - min_len} if op_a['retain'] > min_len else None op2[i2] = {'retain': op_b['retain'] - min_len} if op_b['retain'] > min_len else None if op1[i1] is None: i1 += 1 if op2[i2] is None: i2 += 1 # ... What happens when two users type at the exact same position simultaneously?
Both users submit "insert at position 50" at revision 5. The server receives them in some order — say User A's arrives first. Server applies A's op at revision 6. Then transforms B's op: insert at 50 → insert at 55 (shifted right by A's insertion length). B's op is applied at revision 7. Both inserts appear in the document, ordered by server arrival time. Neither user loses their input.
How does the client know where its own text ended up after transformation?
The server returns the transformed op in the ack message. The client uses this to adjust its local cursor position. If client expected their text at position 50 but the server placed it at 55, the client updates its cursor to 55. This is why the ack includes the final transformed operation, not just a success/fail response.
What's the "intention preservation" problem in OT?
OT can preserve position but not always intent. Classic example: User A types "a" at end of word "cat" intending "cats". User B deletes "t". After OT: "as" — A's 'a' moved to after where 't' was, which is now gone. The text "as" doesn't match A's intent of "cats". This is a known limitation. Google Docs handles this through a more sophisticated OT algorithm (Jupiter protocol) that tracks character identity, not just position.
How do you scale beyond one OT server per document?
True horizontal scaling of a single document is hard — OT requires a total order on operations, which means a single serialization point. Practical approaches: (1) Primary + replicas: primary serializes ops, replicas serve read-only viewers. (2) Section sharding: split a very large document into sections, each with its own OT server (Google Docs does this for large spreadsheets). (3) CRDTs: inherently multi-master, no serialization point, but 50-100x memory overhead per character.
What's the practical limit of users per document?
Google Docs caps live editing at 100 simultaneous editors. At 100 users × 5 ops/sec = 500 ops/sec, each needing transformation against up to 500 concurrent ops. The OT transform is O(1) per op-pair but called O(N²) times in the worst case: 500 × 500 = 250,000 transform operations/second per document. At <1μs per transform, that's 0.25 seconds of CPU/second — feasible on one core. Beyond 100 users, move to a CRDT-based approach.
How does Figma (a CRDT system) handle the memory overhead?
Figma uses a custom CRDT for their vector editing (not text). Vector elements (nodes, properties) are naturally bounded — a design file has thousands of objects, not millions of characters. Text CRDTs like YATA or Automerge store tombstones for every deleted character — a document with 1 million historical characters stores 1 million CRDT nodes even if current content is 100 characters. Figma's CRDT is efficient because design files don't have the deletion-heavy edit patterns of text documents.
How do you handle a user who goes offline for 2 hours and then reconnects?
Client reconnects with its last known revision (say, revision 100). Server has progressed to revision 5,000 during the 2-hour offline period. Server sends all 4,900 operations since revision 100. Client transforms its buffered offline operations against these 4,900 server operations. This is O(offline_ops × server_ops). If the user made 50 edits offline and 4,900 server ops arrived, that's 245,000 transforms — still under 1 second in practice. The client applies the result and submits its transformed ops to the server.
What if the offline user's revision is so old that the operation log is partially GC'd?
If the user's revision falls before the oldest retained operation, the server can't transform incrementally. The server sends the current snapshot and the user's offline edits are presented as a conflict copy (like Dropbox conflict copies), not merged. This is a graceful degradation — no data loss, but manual merge required. In practice, operation logs are retained for 30 days, so this only affects users offline for >30 days.
How do you prevent a slow user from holding up the operation log forever?
Operation log retention is time-based, not revision-based. Ops older than 30 days are GC'd regardless of any user's revision. A user who opens a document and leaves it untouched for 31 days loses the ability to seamlessly merge their changes — but that's an extreme edge case. More practically: the OT server doesn't wait for slow clients; it applies and broadcasts operations as they arrive. Slow clients fall behind and must catch up when they reconnect.
How do you ensure durability — what if the OT server crashes after accepting an operation but before persisting it?
Write-ahead log pattern: the server writes the operation to Postgres before sending the ack to the client and before broadcasting to other users. If it crashes after the DB write but before the ack, the client will resend the op on reconnect — the server detects the duplicate via (doc_id, client_id, revision) and returns the already-committed result. If it crashes before the DB write, the operation is simply lost — the client resends it, it's applied fresh. No durability hole.
What happens to users who received a broadcast of an op that then gets rolled back?
In the write-before-broadcast pattern, broadcasts only happen after the DB write succeeds. A crash before DB write means the op was never broadcast. A crash after DB write but before broadcast: on restart, the new OT server loads from Postgres (which has the op) and can rebroadcast it. Users who reconnect will get the op as part of their catch-up. There's no rollback of a committed op — once it's in Postgres, it's canonical.
How do you handle Postgres being briefly unavailable?
Buffer in-memory for up to 5 seconds with a queue. If Postgres is down >5 seconds, reject new operations with a "temporarily unavailable" error (not a 500 — clients display "saving..." state). The in-memory buffer ensures a brief Postgres hiccup doesn't immediately impact users. On Postgres recovery, flush the buffer in order. With RDS Multi-AZ, failover is <60 seconds — 5 second buffer handles most blips.
Why does Google Docs require a central server — can't OT work peer-to-peer?
The classic OT algorithm (dOPT) does work peer-to-peer but requires O(N²) storage of operation history and has known correctness issues beyond 2 users. The Jupiter protocol (used by Google Docs) requires a central server as the serialization point — it's simpler and correct for N users. Peer-to-peer collaborative editing requires CRDTs (which are commutative by design) or a fully replicated OT log — both significantly more complex. The central server is a pragmatic engineering choice, not a theoretical requirement.
What's the latency penalty of routing all operations through a central server?
With regional deployment (Google has OT servers on 6 continents), the round-trip is 30-100ms depending on user location. Optimistic local application (client shows the character immediately before server confirmation) hides this latency — the user sees their keystroke instantly. The remote user sees it 30-100ms later. This is the same optimistic update pattern used by all real-time collaborative tools.
What does "optimistic local application" mean and what happens on conflict?
Client immediately shows the operation locally without waiting for server ack. If the server transforms the op (because other ops arrived concurrently), the client receives the transformed op in the ack. If the transformation changes the position (e.g., insert shifted from 50 to 55), the client reconciles: rolls back its optimistic application, replays the incoming server ops, then re-applies its own op at the transformed position. This reconciliation happens in under 1ms — imperceptible to the user.

Market Reality: How Indian Startups & Products Solve This
Zoho Writer
🇮🇳 SaaS 15M+ users
How they solve it: Competes directly with Google Docs, built in Java on Zoho's own global infrastructure. Uses Operational Transform (OT) — same as Google Docs — with a custom conflict resolution server. Part of Zoho WorkDrive, documents are stored as structured data where a paragraph is a DB row, not a file chunk.
Stack: Java OT (custom) Zoho WorkDrive
Key difference: Field-level permissions ("only HR can edit salary column") that Google Docs cannot do — enabled by treating a paragraph as a DB row with per-row ACLs, not a blob with file-level permissions.
Notion
🌐 Productivity Widely used by Indian startups
How they solve it: Block-based document model — every block is a CRDT, meaning offline edits always merge without a server round-trip. PostgreSQL for block storage, WebSockets for real-time sync, custom CRDT for offline edits.
Stack: PostgreSQL WebSockets CRDT (custom)
Key difference: Uses CRDTs (not OT) — offline edits always merge without a server round-trip, which is why Notion works seamlessly offline while Google Docs often loses offline edits.
Coda
🌐 Productivity Popular with Indian product teams
How they solve it: Spreadsheet + doc hybrid using Yjs (CRDT library) for real-time collaboration. The "document" is also a data computation engine — formulas can query external APIs like Salesforce live.
Stack: Yjs (CRDT) WebSockets
Key difference: Formulas in Coda docs can query external APIs — the document is both a collaboration surface and a live data computation engine, which standard OT-based systems cannot support.
Zoho Show / Zoho Sheet
🇮🇳 SaaS Zoho suite
How they solve it: Presentation and spreadsheet collaborative editing on the same OT-based architecture as Zoho Writer, but with domain-specific conflict rules tuned per content type.
Stack: Java OT (domain-specific)
Key difference: Cell-level OT for spreadsheets vs paragraph-level OT for documents — the conflict resolution granularity is different for each content domain, requiring separate OT implementations sharing the same server infrastructure.

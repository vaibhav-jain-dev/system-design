{{mustKnow "Log aggregation pipeline has three tiers: Collection (agents/SDKs) → Transport (Kafka) → Storage/Query (Elasticsearch or ClickHouse). Know each tier's failure modes." "Structured logging (JSON) is non-negotiable — unstructured text logs are unparseable at scale and break downstream indexing." "Kafka decouples producers from consumers: handles backpressure, enables replay, and allows multiple consumers (alerting, archival, search) from a single stream." "Elasticsearch inverted index enables full-text search in O(1) per term but costs ~10x more storage than columnar alternatives like ClickHouse." }} {{caveat "Log volume grows faster than your business — a 10x traffic increase can cause 50-100x log growth if debug logging is left on. Always enforce log level controls and sampling at the collection tier, not the storage tier."}} {{goodToKnow "OpenTelemetry is becoming the industry standard for instrumentation. It provides a vendor-neutral SDK that emits logs, traces, and metrics in a unified format — meaning you can switch backends (Datadog → SigNoz → Loki) without re-instrumenting application code."}}
Market Reality: How Indian Startups & Products Solve This
SigNoz
🇮🇳 OSS Observability 18K+ GitHub stars, 3000+ companies
How they solve it: OpenTelemetry-native open source alternative to Datadog, built in India. Architecture: OpenTelemetry Collector → Kafka → ClickHouse. ClickHouse columnar storage makes log queries 10x cheaper than Elasticsearch at scale.
Stack: OpenTelemetry Kafka ClickHouse Go
Key difference: Unified traces + metrics + logs in a single tool (vs ELK for logs + Prometheus for metrics + Jaeger for traces) — one query interface, one retention policy, one bill. Used by Indian unicorns as the cost-effective Datadog alternative at 1/10th the cost.
{{caveat "ClickHouse is excellent for analytical log queries (aggregations, time-range scans) but poor for point lookups by log ID. If your system needs both full-text search and analytics, you may need two storage backends — Elasticsearch for search, ClickHouse for dashboards — which doubles operational complexity."}}
Freshworks
🇮🇳 SaaS Billions of log events/day, 60K+ customer deployments
How they solve it: ELK stack for application logs, Prometheus + Grafana for metrics, custom alerting on top. Log ingestion is tenant-isolated — each customer's logs are in separate Elasticsearch indices with separate retention policies enforced at the Logstash pipeline level.
Stack: Elasticsearch Logstash Kibana Prometheus Grafana
Key difference: Tenant isolation enforced at the Logstash pipeline level (not the query layer) — free tier gets 7-day retention, paid gets 90-day, and the pipeline physically routes to different Elasticsearch indices. Query-layer isolation would allow cross-tenant data leakage bugs; pipeline-layer isolation makes it structurally impossible.
Razorpay / PhonePe
🇮🇳 Fintech PCI-DSS compliant, 1-year mandatory retention
How they solve it: Custom log pipeline for PCI-DSS compliance. Logs are encrypted at ingest, cardholder data is redacted by a custom Logstash filter before storage. PCI-DSS mandates 1-year log retention and real-time alerting on unauthorized access attempts.
Stack: Logstash (custom filter) Elasticsearch Kibana AES-256 encryption
Key difference: PCI-DSS compliance is built into the ingestion pipeline, not bolted on after. Cardholder data never reaches Elasticsearch in plaintext — the Logstash filter runs a regex to detect and redact PAN patterns before the log event leaves the collection tier. Datadog is avoided due to data residency concerns (logs leaving India) and cost at PCI-scale volume.
{{goodToKnow "Retention policies in logging systems are often driven by compliance, not engineering preference. PCI-DSS requires 1-year retention, GDPR requires deletion on request, and SOC 2 requires immutable audit logs. In an interview, always ask about compliance requirements before proposing a retention strategy."}}
Grafana Loki
🌐 OSS Rapidly adopted by Indian startups as ELK alternative
How they solve it: Label-based log aggregation with Prometheus-compatible query language (LogQL). Does not index log content — only indexes labels — making storage 10x cheaper than Elasticsearch. Pairs with Grafana for visualization.
Stack: Loki Promtail Grafana
Key difference: No full-text indexing means log queries require label-based filtering first — fast and cheap for known failure patterns (filter by service + error level), but slow for ad-hoc "find all logs containing X" queries. ELK inverts this: full-text search is fast but expensive; Loki full-text is slow but free.

{{mustKnow "Loki indexes only labels (not log content) — 10x cheaper storage but requires label-based filtering before full-text search. Know when this tradeoff is acceptable." "Sampling (e.g., keeping 1 in 10 debug logs) is the primary cost control lever — a well-tuned sampling strategy can cut storage costs by 80% with minimal observability loss." "Hot/warm/cold tiering is essential: recent logs on SSDs (hot), older logs on HDDs (warm), archived to S3/GCS (cold). Each tier has different query latency and cost characteristics." }} {{caveat "Never log sensitive data (PII, tokens, passwords) — redaction must happen at the collection tier before data enters the pipeline. Retrofitting redaction after logs are stored in Elasticsearch is nearly impossible at scale because you cannot efficiently update or delete individual fields in an inverted index."}}

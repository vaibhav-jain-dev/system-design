How do you guarantee at-least-once delivery without double-sending?
Two-layer approach: (1) Kafka persistence — messages are durable in Kafka for 7 days. Consumer commits offset only after successful delivery — if consumer crashes, Kafka replays from last commit. (2) Redis deduplication — before sending, check SET NX with notification_id + channel. If key exists, message was already sent — skip. TTL=24h. Together: Kafka guarantees delivery attempt, Redis prevents actual double-send.
What if Redis fails during the dedup check?
Fail open: if Redis is unreachable, send the notification anyway and accept the risk of a duplicate. Fail closed (blocking all sends until Redis recovers) is worse — users miss OTP codes, payment confirmations, critical alerts. Most notification systems prioritize delivery over perfect dedup. Log the Redis failure for investigation. The duplicate risk is low if Redis has proper HA (Sentinel or Cluster).
FCM says it delivers messages at-least-once itself — does that mean you can get triplicated notifications?
Yes, technically. Your system sends once (after dedup), FCM delivers at-least-once internally. In practice, FCM internal duplicates are rare (occurs on server-side retries during transient errors). To protect fully: use FCM's collapse_key — FCM holds only the latest message with the same collapse key per device, discarding older ones. Set collapse_key to notification_type + user_id. This reduces FCM-side duplicates to zero — only the latest message per type is delivered.
How do you fan out to 10 million followers without blocking the post action?
Async two-stage fan-out: (1) Post service writes ONE event to Kafka: {type: new_post, user_id: influencer_id, post_id} — completes in <5ms. (2) Fan-out consumer reads event → queries subscriber list in batches of 10K user_ids → for each batch, enqueues to notification Kafka topic. Produces 1,000 Kafka messages (10M / 10K). (3) Delivery consumers process notification batches in parallel → call FCM/APNs. Total: <30 seconds for all 10M notifications with 100 delivery workers.
What if the influencer's subscriber list is 100M users?
Pre-shard the subscriber list. Instead of one query for 100M records, shard the subscribers table by influencer_id % 1000. Fan-out consumer queries all 1000 shards in parallel. Also: for mega-celebrities (100M+ followers), consider a fan-out-on-read hybrid — store the notification event and have clients pull it when they open the app, rather than pushing to 100M devices. This trades notification immediacy for infrastructure savings.
How do you ensure notification ordering for the same user (e.g., "Alice liked your post" before "Alice commented on your post")?
Partition Kafka by user_id (the recipient, not the sender). All notifications for the same user land on the same partition, processed by one consumer in order. This gives per-user ordering. Cross-user ordering (Alice's notifications before Bob's) is undefined — acceptable since users don't see each other's notification feeds. Critical: partition key must be the recipient's user_id, not the triggering event's user_id, to maintain ordering from the receiver's perspective.
How does the priority queue work for OTP vs marketing?
Three separate Kafka topics: notif.high (OTP, payment alerts), notif.medium (likes, comments, follows), notif.low (marketing, promotions). Consumers poll in order: poll high first (timeout=0ms) → if empty, poll medium → if empty, poll low (timeout=100ms to avoid busy-spin). Dedicated consumers for high-priority ensure OTP is never starved by marketing volume.
What SLA would you commit to for OTP delivery?
<10 seconds end-to-end for OTP (notification trigger → user receives SMS). Breakdown: trigger to Kafka: <50ms. Kafka consume: <100ms. Dedup + preference check: <50ms. SMS API call (Twilio): <3s. Total: ~4s. SLA of 10s gives 2.5x buffer. Alert if P99 exceeds 10s. Note: SMS is preferred for OTP over push because SMS works when phone is offline (queued at carrier) and doesn't require app installed.
How do you prevent marketing notifications from delaying OTP by hours during a campaign?
Dedicated consumer pods per priority tier with isolated resource pools. Marketing consumer pods: auto-scale based on low-priority queue lag. OTP consumer pods: fixed minimum capacity of 50 pods, never scaled below that, isolated from marketing autoscaling. Resource isolation via Kubernetes: OTP pods on dedicated nodes (priority class), marketing pods on spot instances. Even if marketing pods are all at max load, OTP pods have guaranteed CPU/network.
How do you prevent notification spam while still guaranteeing OTP delivery?
Two-tier rate limiting: (1) Per-user, per-channel rate limits via Redis token bucket — push: 5/hour, email: 3/hour, SMS: 1/hour. Applied to marketing/social notifications only. (2) Type-based bypass — OTP, payment confirmations, security alerts bypass rate limits entirely. Classification: notification_type enum determines which tier. Also: global rate limit per sender (service) to prevent one service from spamming all users.
How do you implement the Redis token bucket for per-user rate limiting?
Redis key: ratelimit:push:{user_id}. Lua script (atomic): INCR key; if value == 1 → EXPIRE key 3600; if value > limit → return 0 (rate limited). This is a fixed window counter — simpler than sliding window but has boundary burst issue (user can send 5 at 11:59pm + 5 at 12:01am = 10 in 2 minutes). For notifications (not security-critical), fixed window is acceptable. Sliding window via Redis sorted sets for stricter control.
What about quiet hours — how do you respect them without losing the notification?
Buffer notifications during quiet hours rather than dropping them. When a notification would be sent during user's quiet hours (e.g., 10pm-8am local time): store in a scheduled_notifications table with send_at = next allowed time. A scheduler job runs every minute, queries notifications where send_at <= now, enqueues to Kafka for delivery. Use user's timezone from profile. Exception: OTP and critical security notifications always sent immediately — never buffered.
How do you handle FCM going down for an hour?
Circuit breaker on FCM API calls: opens after 5 consecutive 5xx errors (threshold), half-open probe every 60s. During open state: (1) Push notifications queue in Kafka (they're not dropped). (2) Fall back to email for non-push-only users. (3) Alert on-call team. When FCM recovers: circuit closes, Kafka consumer catches up — users get delayed push notifications. For notifications older than 2 hours in DLQ, send as email instead of delayed push.
What about APNs — should you fail over to FCM for iOS?
You can't — APNs and FCM are Apple and Google's respective push systems. iOS devices only receive pushes via APNs; Android via FCM. Cross-fail-over is impossible. Instead: for APNs outage, fall back to email if user has email on file, or queue and deliver when APNs recovers. If APNs outage is extended (>1 hour), prioritize critical notifications (OTP, payments) via SMS as backup channel.
How do you measure notification delivery rate if you can't get individual delivery receipts?
Use proxy metrics: (1) FCM/APNs response codes — success means delivered to provider queue, not device. Track success% as "provider delivery rate." (2) Open rate — if user taps notification, it was delivered. Track tap events via deep link. (3) Unsubscribe rate spike — if users suddenly mass-unsubscribe after a campaign, notifications may have been received as spam (over-delivery). (4) Invalid token rate — rising rate means devices are churning (app uninstall), a leading indicator of stale user base.

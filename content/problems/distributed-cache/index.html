Consistent Hash Ring Implementation Deep Dive
{{code "python" `import hashlib import bisect class ConsistentHashRing: def __init__(self, nodes: list[str], vnodes: int = 150): self.vnodes = vnodes self.ring = {} # position → node_name self.sorted_keys = [] # sorted list of positions for node in nodes: self.add_node(node) def add_node(self, node: str): for i in range(self.vnodes): # Each vnode gets a unique hash by appending vnode index vnode_key = f"{node}:vnode:{i}" position = int(hashlib.md5(vnode_key.encode()).hexdigest(), 16) self.ring[position] = node bisect.insort(self.sorted_keys, position) def remove_node(self, node: str): for i in range(self.vnodes): vnode_key = f"{node}:vnode:{i}" position = int(hashlib.md5(vnode_key.encode()).hexdigest(), 16) del self.ring[position] self.sorted_keys.remove(position) def get_node(self, key: str) -> str: if not self.ring: raise Exception("Ring is empty") key_hash = int(hashlib.md5(key.encode()).hexdigest(), 16) # Find first position >= key_hash (clockwise on ring) idx = bisect.bisect(self.sorted_keys, key_hash) if idx == len(self.sorted_keys): idx = 0 # Wrap around to start of ring return self.ring[self.sorted_keys[idx]]`}}

How does consistent hashing work, and why do you need 150 virtual nodes?
Consistent hashing maps both cache nodes and keys to positions on a circle (hash ring). To find which node owns a key: hash the key, find the nearest node clockwise on the ring. Adding/removing a node only moves keys that were between the old and new node positions — on average K/N keys (1/4 with 4 nodes). Without virtual nodes, real node positions are unevenly distributed — one node might own 40% of the ring, another 10%. 150 virtual nodes per physical node = 600 points on the ring → statistical distribution ensures each node owns approximately 25% ± 5%.
What's the concrete difference in cache invalidation between modulo hashing and consistent hashing?
With 4 nodes using modulo hashing (hash % 4): adding node 5 changes the modulo for ~80% of keys — 80% cache miss rate immediately after adding a node. With consistent hashing: adding node 5 moves only ~20% of keys (those between node 5 and its predecessor on the ring). 80% of cache remains warm — dramatically better during scaling events.
How do you handle rebalancing when a node is added — do you migrate keys?
You don't migrate keys — you let them naturally repopulate. The new node joins the ring. Keys that now map to it will be cache misses → callers fetch from DB and populate the new node. The old node still has the data but won't receive requests for keys that moved — those entries will expire naturally via TTL. Active migration is only needed for large caches where cold-start miss rate would overwhelm the DB. In that case, warm the new node by reading from the previous owner: new_node.set(key, old_node.get(key)) before switching traffic.
How do you implement LRU cache in O(1) for both get and put?
Combine a HashMap (O(1) key lookup) with a doubly-linked list (O(1) front/back removal and insertion). HashMap stores key → node_in_list. List is ordered by recency: most recently used at tail, least recently used at head. GET: lookup in map, move node to tail (O(1) pointer updates). PUT: if exists, update and move to tail. If new and at capacity, remove head node (LRU), insert new node at tail, add to map.
What's the memory overhead of the LRU data structure?
Each entry: 2 pointers in doubly-linked list (prev, next) = 16 bytes + HashMap entry = ~48 bytes overhead per cached item. For 10M items: ~480MB overhead. Plus Redis's own per-key overhead (~72 bytes). Total: ~120 bytes per key for the LRU bookkeeping. At 10M keys this is 1.2GB — significant but acceptable for a cache serving billions of requests.
What happens to LRU performance when the cache is 99% full?
No performance degradation — LRU eviction is always O(1) regardless of cache size. Remove from head of list = 2 pointer updates + 1 HashMap delete. The doubly-linked list head always points directly to the LRU item. This is why LRU is the standard choice: constant-time eviction at any fill level. Contrast with LFU which requires a priority queue — O(log N) eviction.
How do you prevent cache stampede / thundering herd?
Three approaches: (1) Mutex lock: when key expires, only one client fetches from DB (using Redis SETNX as distributed lock with TTL). Others wait, then read from cache after winner populates it. (2) Probabilistic early expiration: re-compute value before it expires — as expiry approaches, probability of refresh increases, so cache is never fully cold. (3) Background refresh: a background thread refreshes keys with TTL <10% remaining, before clients experience a miss.
What's the risk with the mutex lock approach?
Lock holder timeout: if the DB fetch takes longer than the mutex TTL, the lock expires and another client starts a parallel DB fetch — both populate the cache. Mitigation: set mutex TTL = 2× max expected DB fetch time. Also: if lock holder crashes, lock TTL ensures eventual release (no deadlock). Downside: all clients except the winner are blocked waiting — adds P99 latency during cache miss events.
How does probabilistic early expiration work mathematically?
Each cache read checks: p_refresh = exp(-β × remaining_ttl / compute_time). When remaining_ttl is large → p_refresh ≈ 0 (no early refresh). As remaining_ttl → 0 → p_refresh → 1 (always refresh). β controls aggressiveness (β=1 is standard). A single random client re-computes and updates the cache before expiry — key never goes cold. No locking, no blocking, works even with multiple clients simultaneously — each independently rolls the dice.
What's the difference between write-through and write-behind cache?
Write-through: write to cache AND DB synchronously in the same request. Cache and DB always consistent. Extra ~2ms per write (DB write latency). Safe: no data loss on cache crash. Write-behind (write-back): write to cache immediately, queue DB write asynchronously. Low write latency (~0.1ms). Risk: data in cache but not yet in DB — cache crash loses those writes permanently. Write-behind only appropriate when data loss is tolerable (view counts, analytics).
When would write-around be the right choice?
Write-around: bypass the cache entirely on write, go directly to DB. Cache is only populated on the first subsequent read. Best for: (1) Write-heavy data that's rarely read (audit logs, event streams — you write millions but read few). (2) Large batch ingestion where you don't want the cache polluted with data that won't be re-read. (3) Cold data that's accessed once then forgotten — no benefit from caching.
How do you invalidate cache entries when a DB update happens outside the cache layer?
Three strategies: (1) Event-driven invalidation: DB change event (via CDC like Debezium) → Kafka → cache invalidation service → DEL key. (2) Short TTL: let cache entries expire quickly (60s TTL) — accept brief staleness. (3) Version tags: cache key includes version: user:123:v5. On DB update, write cache with new version tag. Old version key expires naturally. Version stored in DB alongside data. At read time, construct key with current version — old version never served again.
How would you handle a hot key getting 90% of traffic to one cache node?
Three-tier solution: (1) Local L1 cache in app servers: top 1000 hot keys cached in-process with 1-5s TTL. Zero network hop. Absorbs 80% of hot key traffic before it hits the cache cluster. (2) Key sharding: viral_video → viral_video:shard_0 ... viral_video:shard_9, spread across 10 nodes. Random shard selection on read. Write to all shards. (3) Read replicas: for genuinely popular keys, add dedicated read replicas for that shard.
What are the trade-offs of key sharding for hot keys?
Pros: distributes read load across N nodes (10 shards = 10x the bandwidth). Cons: (1) Writes must go to all N shards (10x write amplification). (2) Inconsistency window — if shard_3 is updated but shard_7 isn't yet, different clients see different values. (3) Complex invalidation — delete all N shards on expiry. Works best for read-heavy, write-rare keys (static config, feature flags). For frequently updated keys: prefer local L1 cache with short TTL instead.
How do you detect hot keys before they become a production incident?
Redis built-in: redis-cli --hotkeys samples key access frequency using the approximated LFU counter. Returns top-N hot keys. Also: OBJECT FREQ key returns frequency estimate for any key. For proactive detection: instrument the client library to track per-key request counts with a sliding window. Alert when any single key exceeds 10% of total cache traffic — that's a hot key at 10 nodes × 10% = 100% of one node equivalent. Automatic sharding can kick in based on this metric.

OSS & Managed Alternatives
Redis — industry standard; single-threaded event loop; cluster mode for horizontal scale
Memcached — simpler, multi-threaded, no persistence; best for pure string caching with no data structures needed
Aerospike — SSD-native, free tier available; survives restarts without warming; ideal for catalogs too large for Redis RAM budget
Dragonfly — drop-in Redis replacement in C++, multi-threaded; 25× throughput on same hardware; growing adoption for cost reduction
KeyDB — multi-threaded Redis fork; compatible API; active-active replication built-in
AWS ElastiCache — managed Redis/Memcached; automatic failover; most common choice for Indian startups on AWS
Google Memorystore — managed Redis on GCP; used by startups on Google Cloud
Market verdict: Redis is the universal default at every scale. Indian companies at scale (Razorpay, PhonePe, Flipkart) use Redis Enterprise or AWS ElastiCache in cluster mode. The notable trend is Aerospike adoption for large catalogs (Meesho, Myntra) where the catalog is too large for Redis memory budget but too latency-sensitive for PostgreSQL. Dragonfly is gaining interest at Indian startups as a cost-cutting measure — same Redis API, 60% infrastructure savings.

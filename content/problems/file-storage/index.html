{{think "Content-addressable storage (CAS) is the key design pattern: the S3 key IS the hash. Two users uploading the same 100 MB file → same hash → same S3 object → stored once. (whyNot "store file content directly in the files table" "Binary blobs in relational DB tables make backups huge, slow index scans, and can't leverage S3's durability/CDN/parallel download features. Chunked Upload Implementation Deep Dive
{{code "python" `import hashlib import concurrent.futures import requests class FileUploader: CHUNK_SIZE = 4 * 1024 * 1024 # 4 MB def upload_file(self, file_path: str, api_client) -> str: # Step 1: Compute hashes file_hash, chunks = self._compute_hashes(file_path) file_size = sum(c['size'] for c in chunks) # Step 2: Check dedup + get upload session response = api_client.start_upload( filename=file_path.name, file_size=file_size, content_hash=file_hash ) if response.get('dedup'): return response['file_id'] # Already exists! upload_id = response['upload_id'] upload_urls = response['upload_urls'] # Step 3: Upload chunks in parallel (max 5 concurrent) with concurrent.futures.ThreadPoolExecutor(max_workers=5) as executor: futures = [] for i, (chunk, url) in enumerate(zip(chunks, upload_urls)): if not chunk['already_uploaded']: futures.append(executor.submit( self._upload_chunk, chunk['data'], url, i )) concurrent.futures.wait(futures) # Step 4: Complete upload return api_client.complete_upload(upload_id) def _compute_hashes(self, file_path): chunks = [] file_hash = hashlib.sha256() with open(file_path, 'rb') as f: while data := f.read(self.CHUNK_SIZE): chunk_hash = hashlib.sha256(data).hexdigest() chunks.append({'data': data, 'hash': chunk_hash, 'size': len(data)}) file_hash.update(data) return file_hash.hexdigest(), chunks def _upload_chunk(self, data: bytes, url: str, idx: int): # Direct to S3 presigned URL — no auth header needed response = requests.put(url, data=data) response.raise_for_status()`}}

Chunk-Level Dedup & Rabin Fingerprinting Deep Dive
{{think "File-level dedup is great for exact duplicates (same file, different users). (whyNot "content-defined chunking for simplicity" "Rabin fingerprinting (content-defined chunking) gives 20-30% better dedup ratios but adds CPU cost during upload and complexity in the client. How does deduplication actually save money at scale?
At 50M users × 10 GB average = 500 PB logical storage. With 60% dedup efficiency, actual stored bytes = 200 PB. S3 Standard costs $0.023/GB/month → 200 PB = $4.6M/month vs $11.5M without dedup. Chunk-level dedup saves more than file-level: large video files with minor edits share ~95% of chunks.
How do you handle dedup for encrypted files?
Server-side dedup requires the server to see plaintext content hashes. End-to-end encrypted files (where only the client holds the key) can't be deduped at the chunk level — different clients use different encryption keys, so identical plaintext chunks have different ciphertext and different hashes. Convergent encryption solves this: derive the encryption key from the content hash. Same plaintext → same key → same ciphertext → same chunk hash → dedup works. Dropbox uses this for their "block" storage.
What's the security risk of convergent encryption?
Confirmation-of-a-file attack: an attacker who knows the hash of a file (e.g., a copyrighted movie) can check if any user stores it, by sending an upload request with that hash — the server returns a dedup hit without the attacker ever possessing the file. Mitigation: require the client to prove they actually have the file content (provide a random challenge, client responds with HMAC(challenge, content)), not just the hash. This prevents hash-only confirmation attacks.
How does sync work when a device has been offline for 2 weeks?
The device has a stored cursor (last sync sequence number). On reconnect: GET /sync/changes?since={cursor} → server returns all changes since that cursor, paginated at 500 events per page. If the change log retains 30 days of history, any device offline < 30 days can catch up from its cursor. Beyond 30 days, the device must do a full re-sync (re-download folder metadata to rebuild its local state).
What if the change log has 100,000 events to replay?
Paginated replay at 500 events/page = 200 pages. Each page includes only metadata (file_id, event_type, version) — not file bytes. The device only downloads file bytes for files it actually needs (not all changes). Background sync downloads changed files lazily. The metadata catch-up is fast; the content download is selective and prioritized by recency.
How do you handle conflict detection for offline edits?
Each device tracks the server version of each file at last sync time. When uploading an offline edit, include base_version in the request. Server checks: if current version == base_version, apply update (no concurrent edit). If current version > base_version (another device edited while offline), create conflict copy. Version vector clocks provide more nuanced conflict detection for collaborative environments.
How do you store 500 PB durably without losing files?
S3 Standard provides 11 nines (99.999999999%) durability by replicating every object across ≥3 AZs within a region. Additionally: enable S3 Versioning (bucket-level), use S3 Replication to copy to a second region for disaster recovery, enable MFA Delete to prevent accidental bucket deletion. For extra safety: verify chunk SHA256 on download and re-upload from backup if mismatch.
How do you test that durability guarantees are met?
Chaos testing: randomly delete S3 objects and verify automatic recovery. Restore drills: quarterly full-restore test from backup to a clean account. Integrity checks: daily sampling of 0.1% of chunks to verify SHA256 matches metadata. S3's own durability is measured through their redundancy and annual reports — 11 nines means <1 object lost per billion stored per million years.
What's the RTO and RPO for a complete region failure?
With cross-region replication enabled: RPO = replication lag (typically < 15 seconds). RTO = time to failover DNS to secondary region + warm up caches (15-30 minutes). Without active-active setup, the secondary is passive (read-only) until promoted. For Dropbox-scale: active-active across two regions with Route 53 latency-based routing provides sub-minute RTO with zero data loss.
How do you handle very large files (50 GB video files)?
Same chunked upload, but increase chunk size to 32 MB for large files to reduce the number of API calls (50 GB / 32 MB = 1,563 chunks instead of 12,800 at 4 MB). Use S3 Multipart Upload: accepts parts of 5 MB to 5 GB each, max 10,000 parts → supports up to 50 TB per file. Parallel upload with 10 concurrent chunk uploads at 100 Mbps = ~40 seconds for 5 GB.
What about mobile clients with unreliable connections?
Small chunk size (1 MB instead of 4 MB) reduces retry cost on connection drop. Exponential backoff on failed chunks. Upload only on WiFi by default. Progress persistence: save uploaded chunk list to local DB so a phone reboot doesn't restart from zero. Upload session TTL on the server: 7 days instead of 1 hour for mobile clients (since mobile uploads are slower).
How do you stream very large files for preview without downloading entirely?
S3 supports HTTP Range requests: GET /file with Range: bytes=0-1048576 returns just the first 1 MB. For video streaming, the client requests only the chunks needed for the current playback position. Store a chunk manifest (chunk_index → byte_range mapping) in metadata. The player fetches the manifest, then requests only needed byte ranges via CloudFront. This enables seeking without full download — same technique used by video streaming services.
How do you prevent the sync service from becoming a bottleneck at 50M users?
SSE connections are stateless from the server perspective — each connection is just a long-lived HTTP response. Scale horizontally: each sync server node handles 50,000–100,000 concurrent connections (Go coroutines or Node.js event loop). With 1 billion active connections (50M users × ~20 devices), you need 10,000–20,000 sync nodes. User → sync node mapping stored in Redis; when an event fires for user_id, look up their node and forward.
What happens when a sync node crashes?
All SSE connections on that node are dropped. Clients reconnect to any healthy node (load balancer picks one). On reconnect, clients poll /sync/changes?since={cursor} to catch up on missed events. The new node updates Redis: user_id → new_node_id. No data loss — all events are in the change log DB. This is why the polling fallback is important: even if SSE fails, clients catch up.
How do you efficiently fan out a shared folder change to 1000 users?
When a shared folder is modified, look up all users with access (from file_permissions table). Publish one event per user_id to a Redis pub/sub channel (or Kafka partition keyed by user_id). Each sync node subscribes to the channels for users connected to it. The fan-out is O(users_with_access) which is bounded by your maximum share size. For public folders shared with millions, pre-compute the change and store it — subscribers pull rather than push at massive scale.

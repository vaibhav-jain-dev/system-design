Celebrity Merge Implementation Deep Dive
{{code "python" `# Celebrity tweet merge — called at read time during timeline generation import heapq def merge_celebrity_tweets(precomputed_ids: list, user_id: str) -> list: """ Merge pre-computed timeline + celebrity tweets. Time: O(N log K), K = number of celebrities followed. """ celebrity_follows = celebrity_service.get_celebrity_follows(user_id) if not celebrity_follows: return precomputed_ids # no celebrities, skip merge # Fetch recent tweets from each celebrity (parallel Cassandra reads) celebrity_batches = parallel_fetch([ (celeb_id, 50) for celeb_id in celebrity_follows[:50] ]) # Heap-based sorted merge (negate timestamp for max-heap) heap = [] for batch_idx, batch in enumerate(celebrity_batches): for tweet_id in batch: ts = snowflake_to_timestamp(tweet_id) heapq.heappush(heap, (-ts, batch_idx, tweet_id)) for tweet_id in precomputed_ids: ts = snowflake_to_timestamp(tweet_id) heapq.heappush(heap, (-ts, -1, tweet_id)) # Extract sorted, deduplicated merged, seen = [], set() while heap and len(merged) < 200: _, _, tweet_id = heapq.heappop(heap) if tweet_id not in seen: seen.add(tweet_id) merged.append(tweet_id) return merged`}}

Elon Musk posts a tweet with 150M followers — how do you deliver it to all followers without melting the system?
Elon is in the celebrity set (>50K followers), so his tweet skips fanout entirely. It's written to Cassandra and published to Kafka in under 5ms — no Redis writes. When any follower loads their feed, the read-time celebrity merge fetches his last 50 tweets from Cassandra in parallel, merges by timestamp, and returns. Extra latency: ~2ms.
How do you know at read time that a user follows Elon?
The service fetches the user's follow list from Cassandra, then intersects it with the celebrity_ids Redis Set using SINTER. This gives the list of celebrities to merge. Fast even for users with 5K follows.
What if the celebrity_ids Set itself becomes a hotspot — read by 300M users every timeline load?
Cache the celebrity_ids Set in each Timeline Service's local memory, refreshed every 30 seconds from Redis. Local lookup is 10ns vs 0.5ms Redis round-trip. The set rarely changes, so 30s staleness is fine. This removes Redis from the hot path entirely.
What about the SSE stream — do 150M users get a push event for Elon's tweet?
No — SSE push to 150M connections is not feasible. SSE is only for normal fanout accounts (<50K followers). For celebrities, the client polls GET /v1/timeline/home every 30 seconds, which triggers the read-time merge. Celebrity tweets appear within 30 seconds — acceptable per the freshness SLA.
How does Twitter's actual "For You" tab differ — does it still use this celebrity merge approach?
The "For You" algorithmic tab uses a candidate retrieval + ranking pipeline that is largely fanout-on-read for all users, not just celebrities. It retrieves candidates from network graph, engagement graph, and interest-based embeddings, then runs a two-tower neural network for ranking. The celebrity merge problem still applies — celebrity tweet candidates are fetched from Cassandra at query time and scored through the same ranking model.
A user follows 1000 accounts — how do you generate their home timeline in under 200ms?
The pre-computed Redis sorted set for this user already contains the most recent 800 tweet IDs fanned out from the 950 non-celebrity accounts they follow. Reading 20 tweet IDs from the sorted set is O(log N) ≈ 1ms. For the 50 celebrity accounts in their follows, parallel Cassandra reads (50 concurrent queries, 1ms each) return recent tweets in 1ms wall time. The sorted merge of 1000 + 2500 candidates takes < 1ms. Hydrating 20 tweet objects via Redis Hash multi-get adds 1ms. Total: 1 + 1 + 1 + 1 = 4ms for data fetching. Network + serialization + ranking adds ~50ms. Well under 200ms.
What if this user's timeline cache is cold (first login after 2 weeks away)?
Cold start: the Timeline Service runs seed_timeline_cache() — fetch last 20 tweets from each of the 950 non-celebrity follows from Cassandra (950 parallel queries, but batched in groups of 100). Each batch takes ~5ms. Total seed: ~50ms for 950 accounts in 10 batches. We return the first page to the user immediately from the seed result before the full cache is populated, so the user sees content in < 200ms. Background workers continue populating the cache.
How do you avoid a thundering herd when 10M users all come back online simultaneously after a Twitter outage?
Stagger cold-start cache seeding with a jitter queue: instead of seeding all 10M timelines simultaneously, the Timeline Service pushes seed requests to a Kafka topic with a rate limiter (e.g., 100K seeds/s). Each seed takes ~50ms of Cassandra load. At 100K seeds/s, 10M seeds complete in 100 seconds. During this window, uncached timelines fall back to direct Cassandra queries — slower (100-200ms) but functional. The Cassandra cluster is sized with this burst in mind.
How does the cursor work for pagination when the pre-computed timeline and celebrity tweets are merged?
The cursor is the tweet_id of the last tweet on the current page (Snowflake ID ID encodes timestamp). On the next request, the Timeline Service fetches pre-computed IDs with score < cursor_timestamp from Redis, and celebrity tweets with created_at < cursor_timestamp from Cassandra. The merge only considers tweets older than the cursor. This ensures no tweet appears twice across pages, even as new tweets arrive between page loads.
What if a celebrity posts 5 tweets between page 1 and page 2 — do they all appear at the top of page 2?
No. The cursor filters to tweets older than the cursor timestamp. The 5 new celebrity tweets have timestamps newer than the cursor and are excluded from page 2. They will appear when the user refreshes from the top (cursor reset to "now"). This is the correct UX: new tweets appear at the top on refresh, not injected mid-scroll. The same logic applies to normal fanout tweets.
A user unfollows someone — how do you remove their tweets from the timeline immediately?
For normal users (non-celebrity), the unfollow event triggers a background cleanup job: read the unfollowed user's recent tweet IDs from Cassandra (last 800 tweets), then issue a Redis pipeline of ZREM commands on the follower's timeline sorted set. At 100 tweets/s removal rate, removing 800 tweets takes ~8ms in Redis. The timeline immediately reflects the removal on the next page load. For celebrities, no cleanup is needed — their tweets are fetched at read time from the follow list, and since the unfollow removes them from the follow list, they no longer appear in the celebrity merge step.
What about tweets from the unfollowed user that were retweeted by others still in the follow list?
Retweets create new tweet objects with a unique tweet_id. The retweet's tweet_id was fanned out from the retweeter, not the original author. The cleanup job only removes tweet IDs authored by the unfollowed user — retweet objects have author_id = retweeter_id, not original_author_id. So retweets from mutual follows remain in the timeline. This is correct behavior: the user unfollowed the original author, not everyone who retweeted them.
How do you handle a block — the user blocks someone, requiring immediate and retroactive removal?
A block is stricter than an unfollow. On block: (1) immediately add blocker_id + blocked_id to a blocklist Redis Set, (2) trigger the same timeline cleanup job to remove the blocked user's tweets, (3) at read time, filter the merged candidate set against the blocklist before returning results. The read-time filter handles edge cases like retweets of the blocked user's content appearing via third parties. SISMEMBER blocklist at read time is O(1) per tweet candidate.
What if the unfollowed user posted 5000 tweets — removing all from the timeline sorted set is expensive?
The timeline sorted set is capped at 800 tweets. The unfollowed user contributed at most 800 of those 800 entries (and in practice far fewer — they share the 800 slots with everyone else the user follows). The cleanup job only needs to ZREM tweet IDs that are actually in the set, which is a bounded operation. In the worst case (only following one person), it's 800 ZREM commands — a 1ms Redis pipeline operation.
How do you make sure the unfollow cleanup is durable — what if the cleanup worker crashes mid-job?
The unfollow event is written to Kafka before the HTTP 200 is returned to the client. The cleanup consumer processes with at-least-once delivery: it commits its Kafka offset only after the Redis cleanup completes. If the worker crashes mid-cleanup, it replays the event from the last committed offset and re-runs the cleanup. ZREM is idempotent — removing an already-removed member is a no-op. So replaying is safe.
How do you implement chronological vs algorithmic ranking, and what signals does the ML model use?
Chronological ranking requires no ML: the timeline sorted set is ordered by tweet timestamp (Snowflake ID ID score), so ZREVRANGE returns tweets newest-first directly. Algorithmic ranking is a re-ranking layer on top of the same 200-tweet candidate pool: score each candidate with a gradient boosted model or two-tower neural network, sort by score descending, return top 20. Switching between modes is a per-user flag: if ranking_mode = "chronological", skip scoring and return raw ZREVRANGE output.
What features does the ranking model use?
Six feature groups: recency (exponential decay on tweet age), engagement velocity (likes + retweets per hour since posting), author affinity (how often this user has liked/replied to this author, tracked in Redis Hash), content signals (media presence, tweet length, hashtag count), social proof (how many of your other follows liked this tweet), and content embeddings (cosine similarity between tweet embedding and user interest profile embedding). Author affinity and social proof are real-time features served from Redis; embeddings are updated daily via batch ML pipeline.
How do you train the model without introducing a feedback loop that amplifies already-popular content?
Use IPS (Inverse Propensity Scoring) — a technique that weights training examples inversely by the probability the tweet was shown, correcting for the bias that highly-ranked tweets get more engagement simply because they're shown more: Tweets shown to 10M users get weight 1/10M; tweets shown to 1K users get weight 1/1K. This corrects the position bias where highly-ranked tweets get more engagement simply because they're shown more. Additionally, inject a 5% explore budget — show randomly sampled tweets regardless of score — to collect unbiased engagement data for model training.
How do you serve the ranking model at 325K timeline reads per second with < 50ms latency?
Batch inference: the Timeline Service sends a batch of 200 (user_id, tweet_id, features) tuples to the Ranking Service, which runs a forward pass on GPU and returns 200 scores in a single call. GPU throughput: an A10 GPU can process ~10K feature vectors/ms. 200 vectors = 0.02ms compute time. Network round-trip to GPU server: ~2ms. Total ranking latency: ~5ms per timeline request. At 325K RPS, 325K ranking requests/s — deploy a GPU fleet of ~50 A10 instances (each handling ~7K RPS).
What happens if the Ranking Service is down — does the timeline break?
Degrade gracefully to chronological ranking. The Timeline Service has a circuit breaker on the Ranking Service call — if latency exceeds 50ms or error rate exceeds 1%, open the circuit and bypass ranking. Users see chronological feed (still correct and functional) until the Ranking Service recovers. This is a non-critical path: feed availability > feed ranking quality. The circuit breaker resets automatically after 30 seconds.
How do you detect and suppress spam and bot tweets before they spread in the timeline?
Three-layer defense: pre-publish ML classification (block obvious spam before the tweet enters Cassandra), async content moderation (flag and suppress within seconds of publish), and read-time filtering (check blocklist at timeline generation time to suppress previously-missed spam). For most spam, the async layer catches it in 1-3 seconds — before the fanout reaches most followers' timelines.
How does async content moderation work at Twitter scale?
Every tweet published to Kafka is consumed by a Moderation Service that runs a spam/abuse classifier (text + metadata features: URL patterns, account age, tweet rate, follower/following ratio). If the classifier score exceeds a threshold, the tweet's status in Cassandra is set to "suppressed" and its tweet_id is added to a suppressed_ids Redis Set. The entire pipeline runs in < 500ms. Meanwhile, the fanout worker checks suppressed_ids before writing to each follower's timeline — if the tweet is suppressed, fanout is skipped. Tweets already fanned out are cleaned up by a background job.
How do you handle coordinated inauthentic behavior — a botnet of 100K accounts all retweeting the same content?
Velocity anomaly detection: the Moderation Service tracks retweet volume per tweet_id in a Redis sliding window. A tweet receiving 10K retweets in 60 seconds is anomalous (normal viral content grows more slowly). This triggers a coordinated amplification flag — the tweet is queued for human review and its trending contribution is zeroed out. Simultaneously, the 100K bot accounts are flagged by a graph analysis job that detects account clusters with similar creation timestamps, similar following patterns, and synchronized activity. Flagged accounts enter a "low trust" state where their tweets have 0 fanout amplification.
What if a legitimate tweet is incorrectly suppressed by the classifier — how do you handle false positives?
Every suppressed tweet is queued for human review within 24 hours. Users can appeal via the tweet detail page ("This tweet was incorrectly removed"). If a human reviewer overturns the suppression: (1) update tweet status to "active" in Cassandra, (2) remove tweet_id from suppressed_ids Redis Set, (3) trigger retroactive fanout to followers who would have received the tweet. Retroactive fanout uses the same fanout worker with a "delayed" flag. For celebrities, no retroactive fanout is needed — the tweet reappears naturally at next read-time merge.
How do you measure and iterate on classifier accuracy without exposing users to more harmful content?
Use a shadow suppression experiment: run the new classifier in parallel with the current classifier but don't act on the new classifier's decisions — only log them. Compare false positive and false negative rates against human reviewer ground truth labels. Once the new classifier shows 15% fewer false positives with < 5% increase in false negatives on the holdout set, promote it to production. All classifier decisions are logged to a data warehouse for offline analysis and model retraining using the human review labels as supervision signal.

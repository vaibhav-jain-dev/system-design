5M active drivers × 1 update/4s = 1.25M GEO writes/sec. Redis handles 200K+ simple ops/sec per instance with O(log N) GEOADD. PostGIS requires B-tree index update on every write — at 1.25M writes/sec, index contention makes it impossible even on high-end RDS. Redis is single-threaded, lock-free for sorted set operations.
What's the precision of Redis GEO?
52-bit geohash gives ~0.6m accuracy. Sufficient for matching — we don't need centimeter precision to find a driver 200m away. GEORADIUS uses Haversine for distance sorting, which is accurate to within 0.3% for city-scale distances.
How do you handle the driver TTL — what if a driver goes offline without explicitly signing out?
Each driver maintains a heartbeat key (driver:loc:{id}) with 30s TTL, refreshed on every location update. If the app crashes or network drops, the key expires. Before including a candidate in matching, we check EXISTS driver:loc:{id}. GEORADIUS doesn't expire individual sorted set members — the heartbeat key is our liveness signal.
How do you prevent double-booking the same driver for two simultaneous rides?
Before sending an offer, the matching service does SET NX driver:{id}:locked 1 EX 20 in Redis. NX (not exists) is atomic — only one matching instance can acquire it. The lock expires after 20s (offer window + buffer). On accept or reject, the lock is explicitly deleted. This is a single-key lock — Redis guarantees atomicity without Redlock complexity.
What if the matching service crashes after acquiring the lock but before sending the offer?
The 20s TTL handles this. The lock auto-expires. The ride retries matching after a 30s delay (via Kafka scheduled retry). The driver is available again. Net effect: 50-second matching delay for that ride. Acceptable for a crash-recovery scenario.
Why not use database-level locking (SELECT FOR UPDATE) instead of Redis locks for driver assignment?
Database row locks are held for the duration of a transaction. Offering a driver and waiting 15s for their response would hold the lock for 15 seconds — blocking all other operations on that driver row. Redis locks are application-level and don't hold DB connections or transactions. This is why we separate the assignment lock (Redis) from the ride record write (Postgres, milliseconds).
Walk me through surge pricing — how does it actually work end-to-end?
A Kafka consumer reads ride request events and driver location events. Every 5 minutes, a surge calculator groups events by H3 cell (resolution 8, ~0.7km²), computes demand/supply ratio, and writes surge multipliers to Redis (HSET surge:{city} cell_id multiplier). When a rider requests a ride, fare estimate reads the multiplier for their pickup cell from Redis in O(1). The multiplier is shown before confirmation and locked for 5 minutes (the TTL matches the recompute interval).
What happens if a rider's surge multiplier changes during their payment flow?
The fare estimate shown at request time is binding for 5 minutes (matching the surge TTL). If the rider confirms within that window, they pay the quoted surge. If they take longer (edge case — rare), the system re-quotes with the current surge and requires re-confirmation. This is a one-time UX friction for the rare case.
How do you prevent surge gaming — drivers going offline in low-surge areas and coming back online in high-surge areas?
We track driver online/offline events in Kafka with timestamps. If a driver goes offline in cell A and online in cell B within 10 minutes without a completed ride, we apply a 50% bonus cap for that session. Repeated pattern detection (> 3 times in a week) triggers account review. The surge incentive still works — we just don't allow pure manipulation.
How do you scale WebSocket connections for 1M concurrent riders tracking their rides?
WebSocket gateway is a fleet of stateless servers (each handles 10K concurrent connections). Load balancer uses sticky sessions (cookie-based with ALB) to route reconnects to the same server. Each server subscribes to Redis Pub/Sub channels for the rides it's serving. When a driver location update arrives, Redis broadcasts to all subscribers of that ride's channel. Fan-out is O(subscribers) which is always 2 (rider + driver apps) per ride.
Why Redis Pub/Sub instead of Kafka for this fan-out?
Redis Pub/Sub has < 1ms delivery. Location updates fire every 4 seconds — we need low-latency delivery to the rider's map. Kafka has 10–50ms minimum latency (batching). Kafka is appropriate for durable event log (ride state changes). Pub/Sub is appropriate for ephemeral, real-time updates where a dropped message is acceptable (next update arrives in 4s anyway).
What if the Redis Pub/Sub broker crashes mid-ride?
Redis Pub/Sub is in-memory and fire-and-forget — it doesn't persist. If the broker crashes, all subscriptions are lost. WebSocket servers detect the connection drop and re-subscribe to the new primary within 30 seconds (Sentinel failover). During those 30 seconds, riders see a stale map. The app shows a "reconnecting" state. No ride data is lost — ride state lives in Postgres.
How does the system handle a 10x demand spike (e.g., concert ends, major storm)?
Three layers: (1) Auto-scaling — ECS matching workers scale out in 2 minutes on CPU/queue depth metrics. (2) Surge pricing — multiplier increases to attract drivers, reducing demand and increasing supply simultaneously. (3) Virtual queue — if matching service queue depth exceeds 10K rides, new requests are queued with position shown to rider. This prevents thundering-herd on Redis GEO and matching workers.
How fast can the matching service scale out?
ECS Fargate tasks start in ~90 seconds. We keep 20% warm capacity (pre-scaled) at all times. Combined with auto-scaling, we can absorb a 5x demand spike in under 3 minutes. Beyond 5x, surge pricing is the primary demand throttle — not infrastructure scaling.
What's the single biggest bottleneck during a traffic spike?
Postgres — specifically the rides table write path. At 10x normal load, that's ~23K writes/sec to a single primary. We pre-provision RDS with provisioned IOPS (50K IOPS) for this reason. Connection pooling via PgBouncer keeps connections at < 500 to the primary. Beyond that, we shard the rides table by city_id — each city's rides are independent, so cross-city spikes don't compound.

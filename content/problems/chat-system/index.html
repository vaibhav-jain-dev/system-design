How do you handle message delivery when the recipient is offline?
Messages are persisted to DynamoDB immediately after Kafka write. The Presence Service (Redis heartbeat TTL) determines if the user is online. If offline, we write to the offline queue and trigger a push notification via APNs/FCM. When the user reconnects, their client sends its last sync cursor (Snowflake ID message_id). The server queries DynamoDB for all messages > cursor and streams them back over WebSocket.
What if there are 10,000 unread messages when the user comes online?
Paginate the sync. Stream messages in batches of 100 over WebSocket. Render conversations sorted by updated_at so the user sees the most recent conversations first. Background-sync older conversations. The client maintains a local SQLite database for instant rendering of previously loaded messages.
How do you prevent the sync from overwhelming the server?
Rate-limit sync requests per user to 10 batches/sec. Use DynamoDB Limit parameter to cap each query at 100 items. If the user has been offline for 30+ days, consider sending a summary notification instead of syncing all messages. The client can fetch older messages on-demand as the user scrolls.
How does end-to-end encryption work in group chats?
Each group member has a pairwise Signal session with the sender. When sending to a 50-member group, the sender encrypts the message 50 times — once with each member's session key. The server receives 50 ciphertext blobs and routes each to the correct member. The server cannot decrypt any of them.
Isn't encrypting 256 times per message expensive on a mobile device?
AES-256 encryption is hardware-accelerated on modern phones (ARM Cortex has AES-NI). Encrypting a 1KB message 256 times takes < 5ms on an iPhone 12. The real cost is the key management — maintaining 256 Double Ratchet sessions consumes ~50KB of memory per session = 12.5MB for a 256-member group.
What happens when a member is added to or removed from the group?
On add: existing members establish a new pairwise session with the new member via X3DH key exchange. On remove: all remaining members perform a sender key rotation. This ensures the removed member cannot decrypt future messages even if they cached the old sender key. The rotation is triggered by the group admin's client and propagated via the key server.
How do you handle ordering and deduplication at 700K messages per second?
Ordering: Kafka partitions by conversation_id, so all messages in a conversation land in the same partition and are totally ordered. Each message gets a Snowflake ID (timestamp + machine_id + sequence) providing global ordering. Dedup: Client generates a UUID before sending. Server checks idempotency key on DynamoDB conditional write — PutItem with ConditionExpression "attribute_not_exists(message_id)".
What if two messages from different users arrive at exactly the same time?
Snowflake IDs from different Chat Service instances have different machine_id bits, so they're always unique. The timestamp portion provides millisecond ordering. If two messages have the same millisecond timestamp, the machine_id and sequence number break the tie. There's no conflict — both messages are stored with distinct IDs.
How do you handle clock skew across servers?
Snowflake uses NTP-synchronized clocks with a maximum allowed drift of 10ms. If a server detects its clock is drifting beyond threshold, it stops generating IDs and raises an alert. In practice, NTP keeps servers within 1-2ms. For chat, 10ms ordering error is invisible to humans — messages appear in the correct order because human typing speed is measured in seconds, not milliseconds.
How would you design the system for multi-region deployment?
Deploy full stack (WebSocket servers, Chat Service, Kafka, DynamoDB) in 5+ regions. Use DynamoDB Global Tables for multi-region message replication (async, ~1 second lag). Kafka MirrorMaker 2 replicates topic data across regions. Users connect to the nearest region via GeoDNS (Route 53 latency-based routing). Cross-region messages route through Kafka inter-region replication.
How do you handle a conversation where users are in different regions?
The message is produced to the sender's regional Kafka cluster. Kafka MirrorMaker replicates it to the receiver's region. Latency penalty: ~80-120ms for cross-region replication (US-East to EU-West). Total delivery time: ~150-200ms. This is acceptable — users won't notice 200ms vs 50ms in a chat. The alternative (global Kafka) would mean every message pays cross-region latency, even local ones.
What about DynamoDB Global Tables conflict resolution?
DynamoDB Global Tables use last-writer-wins based on timestamps. For messages, there are no conflicts — each message has a unique Snowflake ID as the sort key, so writes never collide. For read_receipts (last_read_message_id), last-writer-wins is correct semantics — you always want the most recent read position. The only risk is with profile updates, where we use conditional writes to prevent stale overwrites.
What are the main failure modes and how do you handle them?
Three critical failure modes: (1) WebSocket server crash — client auto-reconnects in ~2s, syncs missed messages via offline queue. (2) Kafka broker failure — replication factor 3 means 2 brokers can fail without data loss; producers retry on failure. (3) DynamoDB throttling — on-demand capacity auto-scales but has a 2x burst limit; sustained hot partitions trigger adaptive capacity.
What's your disaster recovery plan?
RPO (recovery point objective) = 1 second for messages (Kafka replication lag). RTO (recovery time objective) = 30 seconds for regional failover (DNS propagation). DynamoDB Global Tables replicate to all regions within ~1 second. Kafka MirrorMaker 2 replicates with similar lag. We run chaos engineering monthly — kill a random WebSocket server, fail a Kafka broker, throttle DynamoDB — to verify recovery paths.
How do you prevent a thundering herd when a region recovers?
Jittered reconnection: clients add random delay (0-30 seconds) before reconnecting to the recovered region. Exponential backoff with jitter on WebSocket reconnect prevents all 40M users from connecting simultaneously. The recovered region starts with reduced weight in DNS (10% → 25% → 50% → 100%) over 5 minutes, gradually absorbing traffic.

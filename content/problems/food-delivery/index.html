Why persist state in PostgreSQL, not Redis Deep Dive
{{think "State machine must be persisted in a database, not kept in memory. If the order service crashes after ACCEPTED but before PREPARING, we replay from the last Kafka offset. The consumer is idempotent — replaying a transition on an already-transitioned order does nothing." (whyNot "storing state in Redis only" "Redis is not ACID. A crash between writing state and publishing the Kafka event creates a mismatch between cache and event log. PostgreSQL is the source of truth.") (whatIf "Kafka consumer processes the same event twice" "Each transition uses INSERT ON CONFLICT DO NOTHING keyed on (order_id, from_state, to_state). Duplicates are silently ignored.") }}
{{hint "cancellation at different stages" "PLACED: free cancel. ACCEPTED: free if restaurant hasn't started. PREPARING or later: restaurant fee is non-refundable but delivery fee refunded. PICKED_UP: no refund — food is en route. Interviewers love this edge case."}} {{hint "READY → PICKED_UP trigger" "Partner confirms pickup via the app (QR scan at restaurant). This creates GPS geofence verification — partner must be within 200m of restaurant coordinates to confirm pickup. Prevents fraudulent pickup confirmations."}}
Saga pattern for cross-service payments Deep Dive
{{think "Payment uses the Saga pattern — a chain of local transactions with rollbacks. Order, Payment, and Restaurant Payout are three separate services. If payment succeeds but the restaurant rejects, the saga compensates: refund the customer, cancel the order, notify the restaurant." (whyNot "two-phase commit (2PC) across services" "2PC holds locks until all participants agree. If payment takes 2-5s, that's 350 orders/sec x 5s = 1750 concurrent distributed locks. {{code "python" `# Saga orchestrator for order → payment → restaurant payout class OrderSagaOrchestrator: def execute(self, order_id: str) -> SagaResult: saga_log = [] # Step 1: Charge customer payment_result = payment_service.charge( order_id=order_id, idempotency_key=f"order-{order_id}-payment" ) if not payment_result.success: return SagaResult(success=False, reason="payment_failed") saga_log.append(("payment_charged", payment_result.transaction_id)) # Step 2: Notify restaurant (payout is deferred to delivery) restaurant_result = restaurant_service.accept_order(order_id) if not restaurant_result.success: # Compensate: refund customer payment_service.refund( transaction_id=payment_result.transaction_id, reason="restaurant_unavailable" ) return SagaResult(success=False, reason="restaurant_unavailable") # Step 3: Restaurant payout happens at DELIVERED (not here) return SagaResult(success=True)`}}

{{code "python" `def find_best_partner(order: Order) -> Partner | None: restaurant_lat = order.restaurant.lat restaurant_lng = order.restaurant.lng # Step 1: Geospatial query — Redis GEORADIUS # O(N + log M) where N = candidates returned, M = total indexed partners candidates = redis.georadius( "partners:online", longitude=restaurant_lng, latitude=restaurant_lat, radius=5, # km unit="km", count=20, # cap to 20 nearest sort="ASC", # nearest first withdist=True, withcoord=True ) # candidates = [(partner_id, distance_km, (lat, lng)), ...] # Step 2: Filter by hard constraints eligible = [] for partner_id, distance_km, coords in candidates: partner = db.get_partner(partner_id) if ( partner.status == "AVAILABLE" # not already on delivery and partner.rating >= 4.0 # quality floor and partner.acceptance_rate_30d >= 0.80 # reliability floor and partner.active_orders < max_batch # batch cap (2-3) ): eligible.append((partner, distance_km)) if not eligible: # Widen radius to 8km, relax rating to 3.8 return fallback_match(order, radius=8, min_rating=3.8) # Step 3: Score and rank # score = 0.6 * (1 - norm_distance) + 0.2 * norm_rating + 0.2 * norm_acceptance best = min(eligible, key=lambda p: score(p[0], p[1])) return best[0] def score(partner: Partner, distance_km: float) -> float: # Lower = better (we use min()) distance_score = distance_km / 5.0 # normalize to [0, 1] rating_score = 1.0 - (partner.rating / 5.0) # invert: higher rating = lower score accept_score = 1.0 - partner.acceptance_rate_30d # invert: higher rate = lower score return 0.6 * distance_score + 0.2 * rating_score + 0.2 * accept_score`}}

{{think "Why greedy not optimal? Dijkstra on a city road graph finds the true shortest path but takes 100-500ms per query. At 350 orders/sec that's way too much CPU. {{code "python" `def calculate_eta(order: Order, partner: Partner) -> int: """Returns ETA in seconds from now.""" # Component 1: Remaining restaurant prep time # If order just placed: use restaurant's historical median for this cuisine # If already PREPARING: time elapsed vs estimated prep duration restaurant_prep_remaining = get_prep_remaining(order) # Component 2: Partner travel to restaurant # Haversine straight-line distance × tortuosity factor × traffic multiplier partner_to_restaurant_km = haversine( partner.lat, partner.lng, order.restaurant.lat, order.restaurant.lng ) * 1.3 # road tortuosity factor (straight-line * 1.3 ≈ actual road km) traffic_multiplier = get_traffic_multiplier( zone=order.restaurant.zone, hour=datetime.now().hour ) # 1.0 normal, 1.4 moderate, 2.0 peak partner_to_restaurant_secs = ( partner_to_restaurant_km / PARTNER_AVG_SPEED_KMH * 3600 * traffic_multiplier ) # Component 3: Restaurant to customer travel restaurant_to_customer_km = haversine( order.restaurant.lat, order.restaurant.lng, order.customer.lat, order.customer.lng ) * 1.3 restaurant_to_customer_secs = ( restaurant_to_customer_km / PARTNER_AVG_SPEED_KMH * 3600 * traffic_multiplier ) total_secs = ( restaurant_prep_remaining + partner_to_restaurant_secs + restaurant_to_customer_secs ) # Add 10% buffer for pickup, handoff delays return int(total_secs * 1.1)`}}

10 delivery partners are near the restaurant — which one gets assigned? Walk me through the exact algorithm step by step.
Redis GEORADIUS returns the 20 nearest partners sorted by Haversine distance in ~2ms. We then apply hard constraint filters: status=AVAILABLE, rating ≥ 4.0, 30-day acceptance rate ≥ 80%, active_orders < 3. The remaining candidates are scored: score = 0.6 × normalized_distance + 0.2 × (1 − normalized_rating) + 0.2 × (1 − acceptance_rate). The lowest-scoring partner wins. We issue an optimistic-lock UPDATE on the partner row — whichever concurrent transaction commits first gets the assignment; the loser retries.
What if two partners have nearly identical scores — say score 0.12 vs 0.13?
The 0.01 difference is below noise level of Haversine accuracy. We take the GEORADIUS sort=ASC tiebreak — nearest by straight-line distance. In practice this is rare; a 0.01 score gap corresponds to ~50m distance difference, which is within GPS accuracy anyway.
Should rating carry more weight than 0.2?
Higher rating weight creates a starvation problem — the top-rated partner in a zone gets every order, burning out quickly and inflating their active_orders count. 0.6/0.2/0.2 weighting keeps distance dominant (fast delivery = happy customer) while rating serves as a quality floor enforcer.
What if the winning partner rejects the order?
The matching service immediately re-runs the algorithm excluding that partner. The rejection decrements their acceptance_rate_30d by 1/30d weight. Three consecutive rejections flag the partner for temporary suspension review. The second attempt takes <100ms since Redis state is unchanged.
Does acceptance_rate update synchronously or asynchronously?
Asynchronously via a Kafka rejection event. The matching service publishes the rejection; a separate partner-stats consumer updates the rolling rate in PostgreSQL every 30 seconds in a batch. Real-time matching reads the cached value — 30-second staleness is acceptable here.
A delivery partner's phone dies while they are mid-delivery with the food. How do you detect it and what happens next?
Detection is passive: the partner's location key in Redis has a 60-second TTL that resets on each GPS ping. When the phone dies, pings stop, the key expires, and the partner is automatically removed from the GEORADIUS index. A background scanner runs every 30 seconds querying orders in PICKED_UP state where last_location_update > 90 seconds ago — this fires the PARTNER_UNREACHABLE event.
Why not immediately reassign the order to another partner?
The food is physically with the unreachable partner — reassigning would send a new partner to the restaurant for food that isn't there. The system cannot reassign a PICKED_UP order. Instead: call the partner, notify the customer of delay, start a 5-minute escalation timer. If still unreachable at 5 minutes, escalate to human support.
What refund policy applies in this case?
If the order is not delivered within SLA + 30 minutes, automatic full refund triggers. The partner's account is flagged and their active_orders counter is force-cleared. Refund is issued to original payment method within 3–5 business days or instantly to wallet.
What if the partner reconnects 10 minutes later?
Their app resumes GPS pings. The location service detects the partner_id with an active PICKED_UP order and resumes normal tracking. Customer SSE stream resumes. If a refund was already issued, a reconciliation job flags the order for manual review — we don't auto-reverse the refund.
How do you prevent partners from faking connectivity drops to avoid penalties?
ML anomaly detection on location history. A genuine connectivity drop has irregular gaps. A deliberate disconnect pattern shows precise 60-second gaps coinciding with high-value orders or long-distance deliveries. Three anomalous events in 30 days triggers an account review.
A restaurant delays an order by 20 minutes after the partner is already assigned. How do you update the ETA and notify all parties?
The restaurant updates prep time via PUT /v1/orders/{id}/prep-time with a new_eta body. The order service writes the updated estimated_ready_at to PostgreSQL and publishes an ORDER_DELAYED event to Kafka. The ETA service consumes this event, recomputes total ETA using the new restaurant_prep_remaining value, and publishes an ETA_UPDATED event. Notification service fans this to the customer's SSE stream and sends a push notification within 2 seconds of the restaurant's update.
Does the partner need to be told? They are already on their way to the restaurant.
Yes. The partner app receives an ORDER_DELAYED push notification with the new estimated ready time. If the delay is > 15 minutes and the partner is already within 500m of the restaurant, the app suggests they wait nearby or accept a different order nearby (batch opportunity). The partner is not penalized for arriving late when the restaurant caused the delay.
If the partner accepts a second order while waiting, how does the first order's ETA recalculate?
The batch planner recalculates both ETAs with the partner's current position as the starting point. Order A (delayed) retains its current ETA. Order B gets ETA = A_ready_time + A_delivery_time + travel_B. The customer for Order B is shown the batch-adjusted ETA at order placement — no surprise.
What if the restaurant delays three times on the same order?
After the second delay event, the system sends the customer a cancellation offer — full refund with no penalty. After the third delay, the system auto-cancels and refunds regardless of customer choice, and records a restaurant_reliability_incident. Restaurants with > 3 incidents/week get lower placement in search results.
How is restaurant prep time estimated initially?
A per-cuisine ML model trained on historical delivered_at − accepted_at gaps, segmented by time-of-day and day-of-week. For a new restaurant with no history, we use the cuisine-category median. Median absolute error is ~3 minutes across 10M training samples.
How do you detect and handle fraudulent orders — fake orders placed to harass restaurants, or driver-customer collusion to claim undelivered orders?
Two distinct fraud vectors. Fake orders (harassment): rate limit order placement to 3 orders/hour per customer_id and 5 orders/hour per device fingerprint. New accounts require phone OTP verification before first order. Cancel rate monitoring: if a customer cancels > 40% of orders in 7 days, flag account for review. Collusion (claiming non-delivery): GPS proof-of-delivery — at DELIVERED transition, validate that partner coordinates are within 150m of customer address. Timestamp + GPS coordinates are stored immutably in Cassandra.
What if a partner marks delivery from 500m away (drive-by fraud)?
The order service rejects the DELIVERED transition with HTTP 422 if partner GPS is > 150m from the customer's address coordinates. The partner must physically approach the delivery location. Geofence enforcement is server-side — the app cannot spoof this by faking GPS at the app layer, as location comes from the device OS and is cross-checked against the streaming location history.
What about GPS spoofing apps on rooted Android devices?
The partner app uses SafetyNet Attestation (Android) and DeviceCheck (iOS) to detect rooted/jailbroken devices. Partners on non-attested devices are flagged. Velocity checks also apply — a partner cannot travel 5km in 30 seconds in any location history analysis.
How do you detect a ring of fake customer accounts all ordering from the same restaurant (coordinated harassment)?
Graph analysis: build a bipartite graph of customer → restaurant order edges. A dense cluster (10+ customer accounts all ordering from one restaurant with high cancel rates in 24 hours) is flagged. Shared device fingerprints, IP addresses, or payment methods across the cluster elevates confidence. Automated block at fraud score > 0.85.
Does this fraud detection run synchronously in the order placement path?
No — adding 200ms fraud scoring to every order placement would be unacceptable. Lightweight synchronous checks (rate limit, OTP status) happen inline. Heavy graph analysis runs asynchronously on Kafka ORDER_PLACED events. If fraud is detected post-placement, the order is auto-cancelled and refund reversed within 60 seconds.
Heavy rain causes 90% of delivery partners to go offline simultaneously. How do you handle surge without complete service degradation?
Three simultaneous mechanisms activate. First, surge pricing ramps to 2.5× immediately — the demand/supply ratio jumps to > 4.0 in all zones, triggering maximum multiplier. This sends push notifications to all offline partners advertising the surge rate, historically bringing 35% back online within 10 minutes. Second, order queue depth management: incoming orders are accepted but matching retries are extended from 10 to 30 minutes to give the supply recovery time to work. Third, the app shows realistic ETAs (90–120 minutes) prominently on the restaurant listing page, reducing demand by ~25%.
What if 30 minutes pass and supply still hasn't recovered to serviceable levels?
Graceful demand shedding: The platform temporarily stops showing restaurants in the most supply-starved zones. Customers who try to order see "Limited delivery available in your area" with an estimated recovery time. Orders already in queue keep their position. This is preferable to accepting orders that will never be fulfilled — it protects restaurant ratings and reduces support volume.
How do you define "most supply-starved zones" for the demand shedding decision?
Each zone has a serviceable threshold: minimum available partners to accept new orders (typically 3 per zone). Zones below threshold are marked RESTRICTED in Redis with a 5-minute TTL. The API gateway checks this flag before allowing order placement. The Zone Manager service refreshes zone status every 30 seconds based on live GEORADIUS counts.
Could you pre-position partners before rain by predicting weather-driven demand?
Yes — this is a real operational lever. A weather-demand correlation model trained on historical data shows demand increases 2.5× in rain and supply drops 60%. When weather API forecasts rain in 90 minutes, the platform sends partner incentive notifications (guaranteed minimum per delivery) 60 minutes ahead of onset. This reduces the supply shock from 90% offline to ~50% offline in historically modeled scenarios.
Does pre-positioning require the platform to "pay" partners to be online during rain?
Operationally yes — the surge multiplier combined with rain bonus incentives (flat ₹50 per delivery during weather events) both attracts supply and compensates partners for operating in difficult conditions. The cost is absorbed from the surge pricing revenue — 2.5× multiplier on 10K orders generates ₹5M extra revenue, of which ₹500K covers rain bonuses for 10K deliveries.

Market Reality: How Indian Startups & Products Solve This
Swiggy
🇮🇳 Food Delivery 100M+ orders/month, 300K+ delivery partners across India
How they solve it: Go microservices throughout, Kafka for the order pipeline, Redis GEO for driver location (updated every 3 seconds from the driver app), and a PostgreSQL + DynamoDB hybrid for order records — Postgres for transactional writes, DynamoDB for high-read order history queries. ETA prediction runs a custom XGBoost gradient-boosted model trained on 50M+ historical deliveries.
Stack: Go microservices Kafka Redis GEO XGBoost ETA model Google Maps Distance Matrix
Key difference: ETA prediction is not a distance/speed formula — it's a gradient-boosted model with real-time traffic signals from Google Maps Distance Matrix API injected as features. The model accounts for restaurant prep variability, driver acceptance rate by zone, and time-of-day traffic patterns. The canonical textbook answer (distance ÷ average speed) would be off by 5–10 minutes at Indian traffic density. Accurate ETA is Swiggy's #1 retention driver, not architecture elegance.
Zomato
🇮🇳 Food Delivery (Public) Engineering blog documents production architecture publicly
How they solve it: Go services, Redis Cluster for driver location state, Apache Flink for real-time order aggregation and metrics, Elasticsearch for restaurant search and ranking. Location updates from the driver app flow through a dedicated location service that writes to Redis GEO and emits Kafka events for downstream consumers (ETA service, dispatch service).
Stack: Go Redis Cluster Apache Flink Elasticsearch Kafka
Key difference: Geohash-based Kafka topic partitioning — each geohash cell (roughly 5km²) maps to a dedicated Kafka partition. Driver location updates in Bengaluru don't compete for throughput with Hyderabad events. This partitioning strategy means the dispatch consumer for each city only reads its local partitions, reducing cross-partition coordination and enabling city-level consumer groups to scale independently.
Dunzo
🇮🇳 Quick Commerce (shut down 2024) 10-minute delivery from dark stores
How they solve it: Architecture similar to Swiggy but optimized for sub-10-minute delivery polygons (each dark store serves a ~2km radius). Dark store inventory was managed as a Redis hash per store, not a separate inventory service — stock checks during order acceptance ran against Redis directly for sub-100ms response.
Stack: Redis hash (inventory) Go PostgreSQL Kafka
Key difference: Inventory embedded in Redis as a hash per dark store, not a separate inventory microservice. For a dark store stocking 2,000 SKUs, the entire inventory fits in a single Redis hash that can be read, checked, and decremented atomically using Lua — eliminating a network hop to a separate inventory service on every order acceptance. The tradeoff: Redis becomes the single source of truth for inventory, requiring careful persistence and replication configuration.
ONDC / Namma Yatri Food
🇮🇳 Open Protocol (Government-backed) Decentralized food delivery network
How they solve it: Restaurants publish menus and accept orders via the Beckn open protocol. Any buyer-side app (Paytm, PhonePe, Namma Yatri) can discover restaurants on ONDC and place orders through a standardized API. Each logistics provider (Dunzo, Shadowfax, Loadshare) connects via open logistics API — no single platform owns the driver pool.
Stack: Beckn protocol REST APIs decentralized registry
Key difference: Fully decentralized — there is no Swiggy-equivalent central platform. The order flows from buyer app → ONDC gateway → seller app → logistics provider, with each hop using the Beckn protocol. Driver matching is the logistics provider's responsibility, not the platform's. This eliminates the central dispatch problem entirely but introduces coordination latency between loosely coupled participants.
OSS & Managed Alternatives
OSRM — open-source routing engine (C++); Zomato and Swiggy use OSRM for internal route optimization and ETA baseline; much cheaper than Google Maps at millions of requests/day
Uber H3 — hexagonal geo-indexing library; used by Indian startups for driver search radius queries; H3 hex cells at resolution 8 (~0.7km²) map cleanly to delivery zones and Kafka partition keys
PostGIS — spatial extension for PostgreSQL; used for polygon-based delivery zone management (which restaurants are in delivery range) — simpler than a dedicated geo service for moderate query volumes
Google Maps Platform — most Indian food delivery startups use Google Maps for routing and ETA until they cross ~1M orders/month, at which point OSRM + custom ML becomes cost-competitive
Socket.io + Redis Pub/Sub — standard stack for real-time driver tracking in the customer app; Redis Pub/Sub broadcasts location updates to connected WebSocket sessions per order
Market verdict: Swiggy and Zomato are textbook-quality implementations — their engineering blogs are required reading for this problem. Both converge on Go + Kafka + Redis GEO as the core stack, validating the canonical architecture. The real differentiator at Indian scale is ETA model accuracy, not infrastructure cleverness — a 2-minute ETA improvement drives measurable retention gains. Most Indian food delivery startups at early stage use Google Maps for routing and skip custom geo-indexing entirely until they exceed 1M orders/month, at which point the API cost justifies building on OSRM + H3.
